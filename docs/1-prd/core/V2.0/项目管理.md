## 1. 概述

### 1.1 V1.0 项目配置局限性

V1.0 的项目仅有基本信息（名称、描述、负责人），无法：
- 为项目配置专属的任务类型集合（Issue Type Scheme）
- 为项目配置专属的工作流方案（Workflow Scheme）
- 为项目配置专属的字段行为（Field Configuration Scheme）
- 为项目启用专属的自定义字段
- 为项目配置专属的看板列
- 从项目模板快速创建完整配置

### 1.2 V2.0 核心改进

V2.0 中，**每个项目通过 Scheme 间接引用全局配置**，实现：
- 项目之间可以共享同一套工作流（修改工作流，所有使用它的项目同时生效）
- 项目也可以有自己独立的配置（解耦，不影响其他项目）
- 新项目可以基于模板创建，自动复制全套 Scheme 配置
- 管理员可以全局管理所有 Issue Type / 工作流 / 自定义字段

### 1.3 Jira 的项目 Scheme 映射

Jira 中每个项目关联的 Scheme 列表：

| Scheme | 作用 |
|--------|------|
| Issue Type Scheme | 允许创建哪些类型 |
| Workflow Scheme | 每种类型用哪个工作流 |
| Issue Type Screen Scheme | 每种类型用哪个屏幕方案 |
| Field Configuration Scheme | 每种类型的字段行为 |
| Permission Scheme | 权限控制 |
| Notification Scheme | 通知规则 |
| Issue Type Screen Scheme | 屏幕方案 |
| Priority Scheme | 优先级方案 |

LEST V2.0 实现（优先实现核心 Scheme，Permission 和 Notification 后续迭代）：

| Scheme | 说明 | 核心度 |
|--------|------|--------|
| Issue Type Scheme | 项目允许的 Issue Type | P0 |
| Workflow Scheme | 类型→工作流映射 | P0 |
| Issue Type Screen Scheme | 类型→Screen 方案映射 | P0 |
| Field Configuration Scheme | 类型→字段行为 | P0 |
| Priority Scheme | 优先级集合 | P1 |
| Board Scheme | 看板配置 | P1 |

```
用户选择创建方式：
  │

  │     ▼

  │     ▼
  │   模板 → 复制全套 Scheme 配置

  │     - 复制 Board 配置
  │     - 复制自定义字段定义
  │     │
  │     ▼

  │     │
  │     ▼
  │   创建完成
  │
  └── 空白创建
        │
        ▼
      填写项目基本信息
        │
        ▼
      选择 Issue Type Scheme（默认/自定义）
        │
        ▼

        │
        ▼
      选择 Field Configuration Scheme
        │
        ▼

        ▼
```

## 4. 项目模板体系

### 4.1 模板创建流程

```
管理员创建模板：
  │
  ▼
1. 填写模板基本信息（名称、分类、图标）
  │
  ▼
2. 配置 Issue Type Scheme
  - 选择启用的 Issue Type
  - 可新增项目专属的 Issue Type
  │
  ▼
3. 配置 Workflow Scheme
  - 导入已有工作流 或 新建工作流
  - 为每种类型选择工作流
  │
  ▼
4. 配置 Screen Scheme
  - 导入已有 Screen 或 新建 Screen
  - 配置每个操作的 Screen
  │
  ▼
5. 配置 Field Configuration Scheme
  - 选择全局自定义字段
  - 配置每个类型的字段行为
  │
  ▼
6. 配置看板默认布局
  │
  ▼
7. 模板发布（进入 project_template 表 + project_template_content）
```

| 模板名称 | 分类 | 包含配置 |

## 5. Scheme 层级关系

### 5.1 完整 Scheme 引用链路

```
project
  │
  ├── issue_type_scheme_id ───→ issue_type_scheme
  │                               └── 包含 issue_type_ids[]
  │                                     │
  │                                     └── 每个 issue_type 有：
  │                                           - default_workflow_id
  │                                           - default_screen_id
  │
  ├── workflow_scheme_id ───→ workflow_scheme
  │                            └── 包含 (issue_type_id → workflow_id) 映射
  │                                  覆盖 Issue Type 的默认工作流
  │
  ├── screen_scheme_id ───→ issue_type_screen_scheme
  │                            └── 包含 (issue_type_id → screen_scheme_id) 映射
  │                                  覆盖 Issue Type 的默认 Screen
  │
  ├── field_config_scheme_id ───→ field_config_scheme
  │                                  └── 包含 (issue_type_id → field_config_id) 映射
  │                                        覆盖全局字段配置
  │
  └── board_id ───→ board
                      └── board_columns[]
```

### 5.2 配置优先级

```
项目自定义 > 项目模板 > 全局默认
```

| 配置来源 | 优先级 | 说明 |
|---------|-------|------|
| 项目自定义 | 最高 | 在项目设置中手动修改的配置 |
| 项目模板 | 中 | 从模板创建时继承的配置 |
| 全局默认 | 最低 | 系统预设的默认值 |

```sql
CREATE TABLE permission_scheme (
```

### 7.4 权限检查逻辑

```java
@Service
public class PermissionService {

    public boolean hasPermission(Long userId, Long projectId, String permissionKey) {
        // 1. 检查用户是否为系统管理员（系统管理员拥有所有权限）
        if (userService.isAdmin(userId)) return true;

        // 2. 获取用户在项目中的角色
        ProjectMember member = projectMemberService.getMember(projectId, userId);
        if (member == null) return false;

        // 3. 获取项目的 Permission Scheme
        Project project = projectService.get(projectId);
        PermissionScheme scheme = permissionSchemeService.get(project.getPermissionSchemeId());

        // 4. 检查该角色是否拥有该权限
        return scheme.hasPermission(member.getRole(), permissionKey);
    }

    public void checkPermission(Long userId, Long projectId, String permissionKey) {
        if (!hasPermission(userId, projectId, permissionKey)) {
            throw new PermissionDeniedException("您没有执行此操作的权限");
        }
    }
}
```

## 8. Notification Scheme（通知方案）

每个项目可配置独立的 Notification Scheme，定义谁在什么情况下收到通知。

### 8.1 Notification Scheme 表

```sql
CREATE TABLE notification_scheme (
  scheme_id BIGINT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(128) NOT NULL,
  description TEXT,
  project_id BIGINT COMMENT 'NULL 表示全局方案',
  is_default TINYINT DEFAULT 0,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME ON UPDATE CURRENT_TIMESTAMP
);

CREATE TABLE notification_scheme_event (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  scheme_id BIGINT NOT NULL,
  event_type VARCHAR(64) NOT NULL COMMENT 'issue_created / issue_assigned / issue_transitioned 等',
  notification_type VARCHAR(32) NOT NULL COMMENT 'user / group / role / role_based',
  user_id BIGINT COMMENT '指定用户ID',
  group_id BIGINT COMMENT '指定用户组ID',
  project_role VARCHAR(32) COMMENT '项目角色',
  role_based_type VARCHAR(32) COMMENT 'assignee / reporter / lead / watcher / project_lead 等',
  channel VARCHAR(32) COMMENT '站内信 / 邮件 / plugin:{channelId}',
  UNIQUE KEY uk_scheme_event_notification (scheme_id, event_type, notification_type, COALESCE(user_id, group_id, project_role))
);
```

### 8.2 通知事件类型

| event_type | 说明 | 默认通知对象 |
|-----------|------|-----------|
| issue_created | Issue 创建 | 项目管理员 |
| issue_updated | Issue 更新 | 负责人、关注者 |
| issue_assigned | Issue 分配 | 负责人 |
| issue_unassigned | Issue 取消分配 | 原负责人 |
| issue_transitioned | Issue 状态变更 | 负责人、关注者 |
| issue_commented | Issue 评论 | 被@提及者、负责人 |
| issue_deleted | Issue 删除 | 项目管理员 |

## 11. Issue Security Level（项目安全级别）

> **状态**: ⚠️ V2.0 PRD 中缺少此节，V3.0 已补充完整设计。参见 [任务管理 V3.0 PRD](../V3.0/任务管理.md)。

### 11.1 功能说明

Issue Security Level（项目安全级别）控制用户对特定 Issue 的可见性。当一个 Issue 被设置了安全级别后，只有拥有该级别权限的用户才能查看该 Issue。

### 11.2 Jira 对齐

| Jira 功能 | LEST 支持 | 说明 |
|-----------|---------|------|
| Security Levels | ✅ | 每个项目可配置多个安全级别 |
| Default Security Level | ✅ | 新建 Issue 时的默认安全级别 |
| Grant 表 | ✅ | 用户/组 → 安全级别映射 |
| Per-Issue Security | ✅ | 每个 Issue 可单独设置安全级别 |

### 11.3 数据库设计

```sql
-- 安全级别
CREATE TABLE `security_level` (
    `id`                BIGINT PRIMARY KEY AUTO_INCREMENT,
    `name`              VARCHAR(64) NOT NULL COMMENT '安全级别名称',
    `description`       VARCHAR(256),
    `level_order`       INT NOT NULL DEFAULT 0 COMMENT '级别顺序（数字越大越高）',
    `project_id`        BIGINT NOT NULL COMMENT '所属项目，NULL 表示全局级别',
    `created_by`        BIGINT NOT NULL,
    `created_at`        DATETIME DEFAULT CURRENT_TIMESTAMP,
    `updated_at`        DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- 安全级别授权
CREATE TABLE `security_level_grant` (
    `id`                BIGINT PRIMARY KEY AUTO_INCREMENT,
    `security_level_id` BIGINT NOT NULL,
    `grant_type`        VARCHAR(16) NOT NULL COMMENT 'USER/GROUP/PROJECT_ROLE',
    `grantee_id`        BIGINT NOT NULL,
    FOREIGN KEY (`security_level_id`) REFERENCES `security_level`(`id`)
);
```

### 11.4 用户故事

| ID | 用户故事 | 验收标准 |
|----|---------|---------|
| US-SEC-001 | 查看安全级别列表 | 安全级别列表显示名称、顺序、已授权用户数 |
| US-SEC-002 | 授权用户/组访问安全级别 | 支持按用户/组授权，授权后对应用户可看到该级别 Issue |
| US-SEC-003 | 为 Issue 设置安全级别 | Issue 编辑页面可选择安全级别，无权限用户无法看到 |

### 11.5 API 设计

| 接口 | 方法 | 说明 |
|------|------|------|
| `/project/{id}/security-levels` | GET/POST | 安全级别列表/创建 |
| `/project/{id}/security-levels/{levelId}` | PUT/DELETE | 更新/删除安全级别 |
| `/project/{id}/security-levels/{levelId}/grants` | GET/POST/DELETE | 授权管理 |
| `/issue/{id}/security-level` | GET/PUT | Issue 安全级别读取/更新 |

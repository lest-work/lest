## 1. 为什么需要插件架构

### 1.1 没有插件生态的问题

如果所有功能都放在核心（LEST Core），会出现以下问题：

| 问题 | 影响 |
|------|------|
| 核心越来越臃肿 | 每个团队需求不同，90% 的功能对大多数用户无用 |
| 开发维护成本高 | CI/CD、工时统计、IDE 集成等都需要核心团队维护 |
| 更新风险高 | 修改一个模块可能影响所有用户 |
| 第三方无法参与 | 好的功能只能等官方开发 |
| 难以私有化定制 | 企业有特殊需求只能 fork 代码 |
| 与 Jira 的差距 | Jira 有 8000+ 插件，LEST 如果没有插件生态就永远只是"简化版 Jira" |

### 1.2 正确的架构决策

**核心（Core）只保留"必不可少"的功能**。其他一切都可以是插件：

```
LEST Platform
│
├── Core（核心层）— 最小化，只做平台基础设施
│   ├── 认证授权（登录/Token/权限）
│   ├── 项目与 Issue 管理（CRUD + Scheme 架构）
│   ├── 工作流引擎（流转执行）
│   ├── 通知框架（通知发送框架，不含具体渠道）
│   ├── 插件系统（安装/启用/禁用/SDK）
│   └── 开放平台（API/Webhook 框架）
│
├── Official Plugins（官方插件）— 官方开发和维护
│   ├── lest-worklog（工时管理）
│   ├── lest-cicd（CI/CD 集成）
│   ├── lest-wakatime（编码追踪）
│   ├── lest-devops（DevOps 集成）
│   ├── lest-im-feishu（飞书集成）
│   ├── lest-im-dingtalk（钉钉集成）
│   ├── lest-im-wecom（企业微信集成）
│   ├── lest-ai（AI 服务）
│   ├── lest-report（报表中心）
│   ├── lest-meeting（敏捷会议）
│   └── lest-release（发布管理）
│
└── Community Plugins（社区插件）— 社区开发
    ├── Jira Sync（Jira 双向同步）
    ├── GitHub Integration（GitHub 深度集成）
    ├── GitLab Integration（GitLab 深度集成）
    ├── Linear Sync（Linear 同步）
    ├── ZenTao Import（禅道数据导入）
    ├── Excel Import（Excel 导入）
    ├── Sla Timer（SLA 倒计时）
    └── ...（任何第三方可开发的功能）
```

### 1.3 核心 vs 插件的划分原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **核心依赖原则** | 如果功能是其他所有功能的前置依赖，必须在核心 | 认证、项目、Issue、工作流引擎 |
| **数据依赖原则** | 如果功能的数据模型是其他功能的数据基础，必须在核心 | Issue 表、Project 表 |
| **扩展性原则** | 如果功能有无数种实现方式，放插件 | IM 通知（飞书/钉钉/企微完全不同）|
| **稳定性原则** | 如果功能变更频率低、API 稳定，放核心 | 工作流引擎 |
| **竞争性原则** | 如果功能有多个互斥实现，放插件 | AI 模型（Qwen/Kimi/GPT）|
| **可插拔原则** | 如果功能可以被完全移除而不影响核心运行，放插件 | WakaTime、工时统计 |
| **中国特色原则** | 中国特有的集成（钉钉/企微/飞书），因为 Jira 没有，必须是插件 | IM 集成套件 |

## 2. Core 核心层设计

### 2.1 Core 包含的模块

| 模块 | 说明 | 为什么在 Core |
|------|------|------------|
| **认证授权** | 登录、Token、OAuth2、基础权限 | 所有功能的前置依赖 |
| **项目管理** | Project CRUD、成员、角色 | 所有工作的顶级容器 |
| **Issue 管理** | Issue CRUD、Scheme 架构、自定义字段 | 核心中的核心 |
| **工作流引擎** | 状态流转、条件、校验、动作 | 平台可定制化的核心能力 |
| **通知框架** | 通知模板、事件总线、发送框架 | 需要统一事件源 |
| **插件系统** | 插件生命周期、SDK、扩展点 | 平台的扩展基础设施 |
| **开放平台** | API Key、Webhook 框架 | 外部集成的基础 |
| **AI 框架** | AI 网关、多模型路由（不含具体模型） | 需要统一的 AI 调用规范 |

### 2.2 Core 不包含的模块

| 原 V1.0 模块 | 迁移到 | 原因 |
|-------------|--------|------|
| 工时管理 | `lest-worklog` 插件 | 可选功能，非所有团队都需要 |
| CI/CD 集成 | `lest-cicd` 插件 | Jenkins/GitLab CI/GitHub Actions 各不同 |
| WakaTime 集成 | `lest-wakatime` 插件 | 可选功能 |
| 飞书集成 | `lest-im-feishu` 插件 | 中国特色，Jira 没有 |
| 钉钉集成 | `lest-im-dingtalk` 插件 | 中国特色 |
| 企业微信集成 | `lest-im-wecom` 插件 | 中国特色 |
| 敏捷会议 | `lest-meeting` 插件 | 不是所有团队都用会议功能 |
| 发布管理 | `lest-release` 插件 | 不是所有项目都有正式发布 |
| 团队绩效 | `lest-report` 插件 | 报表类功能放插件更灵活 |
| AI 服务（具体实现）| `lest-ai` 插件 | 具体 AI 模型实现是插件，框架在 Core |

### 3.5 官方插件详情设计

#### 3.5.1 `lest-worklog` — 工时管理插件

**功能：**
- 工时记录（按天/按任务）
- 工时统计（个人/团队/项目）
- 工时报表（柱状图/折线图）
- 工时审核（可选）
- 工时告警（超过预估 80%/100%）

**扩展点使用：**
- `TASK_FIELD`：添加"工时"Tab
- `ACTION_BUTTON`：任务操作栏添加"记工时"按钮
- `WORKFLOW_POST_FUNCTION`：工作流完成后触发工时提醒
- `DASHBOARD_WIDGET`：仪表盘显示工时汇总

**数据库表前缀：** `wl_`

**依赖：** Core（无其他插件依赖）

---

#### 3.5.2 `lest-cicd` — CI/CD 集成插件

**功能：**
- Jenkins 集成：构建触发、状态展示、日志查看
- GitLab CI 集成：Pipeline 状态、MR 关联
- GitHub Actions 集成：Workflow 状态、Commit 关联
- 构建失败自动创建 Bug

**扩展点使用：**
- `WEBHOOK_HANDLER`：接收 CI Webhook
- `ACTION_BUTTON`：任务详情添加"查看构建"按钮
- `TASK_TAB`：Dev Panel（构建、MR、Commit）
- `EVENT_LISTENER`：监听 Commit 推送事件

**数据库表前缀：** `ci_`

**依赖：** Core

---

#### 3.5.3 `lest-wakatime` — 编码追踪插件

**功能：**
- 兼容 WakaTime API（WakaTime 插件直接对接）
- 编码时间统计（按项目/语言/编辑器）
- 活动热力图
- 任务关联（按文件路径匹配）
- 为绩效模块供数

**扩展点使用：**
- `API_PLUGIN`：提供 WakaTime 兼容 API 端点
- `DASHBOARD_WIDGET`：编码时间仪表盘
- `PROJECT_TAB`：项目编码统计 Tab

**数据库表前缀：** `wt_`

**依赖：** Core

---

#### 3.5.4 `lest-im-feishu` / `lest-im-dingtalk` / `lest-im-wecom` — IM 集成套件

**功能：**
- 消息通知推送（任务分配/状态变更/评论 @）
- 审批流集成（钉钉/飞书原生审批）
- 组织架构同步（部门/成员）
- 机器人卡片消息

**扩展点使用：**
- `NOTIFICATION_CHANNEL`：实现通知渠道（飞书/钉钉/企微）
- `WEBHOOK_HANDLER`：接收 IM 平台的 Outgoing Webhook
- `MENU_ITEM`：系统菜单添加 IM 设置
- `SCHEDULED_JOB`：定时同步组织架构

**数据库表前缀：** `im_feishu_` / `im_dingtalk_` / `im_wecom_`

**依赖：** Core

**特殊说明：** 三个 IM 插件相互独立，可单独安装，也可同时安装。

## 4. 插件开发规范

### 4.1 插件命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 官方插件 | `lest-{功能}` | `lest-worklog`, `lest-cicd` |
| 社区插件 | `{功能简名}-lest` | `jira-sync-lest`, `feishu-lest` |

### 4.2 插件必需包含

每个插件必须包含以下内容：

```json
{
  "id": "lest-worklog",
  "name": "工时管理",
  "name_en": "Worklog Management",
  "version": "1.0.0",
  "description": "工时记录、工时统计、工时报表",
  "author": "LEST Team",
  "author_email": "plugins@lest.dev",
  "license": "MIT",
  "categories": ["project-management"],
  "platformVersion": ">=1.0.0 <3.0.0",
  "isOfficial": true,
  "isFree": true,
  "extensionPoints": [
    { "type": "TASK_TAB", "id": "worklog-tab" },
    { "type": "ACTION_BUTTON", "id": "log-work-button" }
  ],
  "permissions": [
    "task:read",
    "task:write",
    "worklog:manage"
  ],
  "dependencies": [],
  "resources": {
    "frontend": true,
    "backend": true,
    "database": true
  }
}
```

### 4.3 插件评分机制

| 评分维度 | 权重 | 说明 |
|---------|------|------|
| 功能完整性 | 30% | 功能是否完整实现 |
| 稳定性 | 25% | 是否稳定无 BUG |
| 易用性 | 20% | 配置和使用是否简单 |
| 文档质量 | 15% | 文档是否完整清晰 |

## 5. V1.0 与 V2.0 插件规划总览

### 5.1 V1.0 发布时

**Core：** 完整包含所有核心功能（向后兼容，不强制插件化）

| 交付方式 | 模块 |
|---------|------|
| **Core 内置** | 认证、项目、Issue、工作流、工时、CI/CD、WakaTime、AI、会议、发布、通知（IM）、绩效 |
| **官方插件** | 无（V1.0 所有功能内置，V2.0 开始拆分）|

> **说明**：V1.0 不强制插件化，所有功能内置是为了快速验证产品。同时将插件系统作为 V1.0 的扩展能力。

### 5.2 V2.0 发布时

**Core：** 精简到最小化

| 交付方式 | 模块 |
|---------|------|
| **Core** | 认证、项目、Issue、工作流引擎、通知框架、插件系统、开放平台 |
| **官方插件（V1.0→插件）** | 工时管理、CI/CD、WakaTime、AI 服务、IM 集成（飞书/钉钉/企微）|
| **官方插件（V2.0 新增）** | 报表中心、敏捷会议（增强）、发布管理（增强）、Jira 同步、禅道导入 |

### 5.3 版本对比

| 对比项 | V1.0（内置） | V2.0（插件化） |
|--------|-------------|--------------|
| Core 复杂度 | 高（所有功能耦合）| 低（最小化核心）|
| 插件数量 | 0 | 12+ |
| 安装灵活度 | 低（全部安装）| 高（按需安装）|
| 维护成本 | 高 | 低 |
| 第三方参与 | 困难 | 容易 |
| 私有化定制 | 困难 | 容易 |
| Jira 对齐 | 功能对齐 | 功能 + 架构对齐 |
| 插件市场 | 无 | 有 |

### 5.4 迁移策略

#### 5.4.1 V1.0 → V2.0 平滑迁移

```
阶段 1：V2.0 初期
- Core 保持内置所有 V1.0 功能（向后兼容）
- 插件系统上线但非必须使用
- 官方插件可选安装，不安装则使用内置版本

阶段 2：V2.0 中期
- 新功能全部以插件形式发布
- V1.0 内置功能逐步拆分为插件（每个 Sprint 一个）
- 提供"一键插件化"工具，自动将内置功能转为插件

阶段 3：V2.0 后期
- Core 精简到最小化
- 所有功能均为插件
- 插件市场成为主要分发渠道
```

#### 5.4.2 迁移检查清单

```markdown
- [ ] 确认 Core 只包含核心依赖模块
- [ ] 所有业务模块已转为插件
- [ ] 插件系统支持插件间依赖声明
- [ ] 提供插件安装向导（从内置迁移到插件）
- [ ] 插件卸载后数据保留机制
- [ ] 插件市场上线
- [ ] 所有官方插件已迁移并测试
```

## 1. 为什么是 Jira 方案

### 1.1 灵活性决定可用性

| 场景 | Jira 能做到 | V1.0 做不到 |
|------|------------|-------------|
| 运维团队想要"故障单"类型 | 新增 issue type + 专属字段 | ❌ 只有 4 种固定类型 |
| 需求评审流程：提交→评审→通过→开发 | 自定义状态 + 工作流路径 | ❌ 只有 todo→进行中→已完成 |
| Bug 需要"严重程度"和"复现步骤" | 自定义字段 + 专属表单 | ❌ 无自定义字段 |
| 不同项目用不同状态集合 | 每个项目独立配置 | ❌ 全局固定 3 种状态 |
| 紧急项目跳过审核直接上线 | 工作流条件跳过 | ❌ 无条件判断 |

> **没有灵活性，就没有真正的项目管理系统**。用户的需求各不相同，固定字段/状态/流程的系统只能在早期快速启动，但很快就会成为瓶颈。

### 1.2 Jira 的 4 层方案映射（Jira 的核心灵活性来源）

Jira 的灵活性来自一套**多对一间接映射**：

```
项目 (Project)
    │
    ├── Issue Type Scheme（定义允许创建哪些类型）
    │       │
    │       └── Issue Type（Bug / Story / Epic / Task）
    │             │
    │             ├── Workflow Scheme（该类型用哪个工作流）
    │             │       │
    │             │       └── Workflow Definition（状态 + 流转规则）
    │             │
    │             ├── Issue Type Screen Scheme（该类型用哪个屏幕方案）
    │             │       │
    │             │       └── Screen Scheme（每个操作对应哪个 Screen）
    │             │             │
    │             │             └── Screen（哪些字段在哪个 Tab）
    │             │
    │             └── Field Configuration Scheme（字段行为配置）
    │                     │
    │                     └── Field Configuration（每个字段：隐藏/显示/必填）
    │
    └── Custom Field Context（字段的作用范围）
            │
            └── Custom Field Definition + Custom Field Option + Custom Field Value
```

**这套架构的核心思想**：每一个"什么角色看什么"都通过 Scheme 间接映射，而不是硬编码。从而实现：
- 一个字段可以同时存在于多个类型中，每个类型对它的要求不同
- 一个工作流可以同时用于多个类型
- 一个 Screen 可以被多个类型共用

### 1.3 LEST 的方案选择

LEST V2.0 完整采用 Jira 方案，并根据 SaaS 多租户特点优化：

| 维度 | Jira 实现 | LEST 实现 | 改进点 |
|------|-----------|-----------|--------|
| 自定义字段存储 | 独立表 + 多列（STRING/NUMBER/TEXT） | 同 Jira + JSON 列兜底 | 支持快速查询 |
| 字段选项 | 独立表（customfieldoption） | 同 | - |
| 字段上下文 | 全局 + 项目级 | 同 + 增加"全局/项目/模板"三级 | 支持项目模板 |
| 字段配置方案 | fieldconfigscheme | 同 | 增加 Issue Type 级别配置 |
| Screen | 独立表，字段+Tab 布局 | 同 | - |
| Screen Scheme | screen→操作 | 同 | - |
| Issue Type Screen Scheme | 类型→screen scheme | 同 | - |
| 工作流 | XML 描述符 | JSON 描述符（更易读） | 前端可解析展示 |
| 工作流条件/校验/动作 | Java 代码 | JSON 规则引擎（可配置） | 无需写代码 |
| Issue Type Scheme | 项目允许的类型 | 同 | - |
| 工作流方案 | 工作流→项目 | 同 | - |

## 2. 核心概念与层级

### 2.1 概念层级图

```
平台级
│
├── 全局 Issue Type 模板（story / task / bug / epic / tech_debt）
├── 全局自定义字段定义（severity / sprint / team）
├── 全局状态库（todo / in_progress / reviewing / testing / done / ...）
├── 全局工作流定义
└── 全局 Screen 模板
        │
        ▼
项目级（每个项目独立）
│
├── Issue Type（引用全局模板，可覆写）
│     │   每个类型有独立的：
│     │   - Workflow Scheme（用哪个工作流）
│     │   - Issue Type Screen Scheme（用哪个屏幕方案）
│     │   - Field Configuration Scheme（字段行为）
│     │
│     ├── Story 类型
│     ├── Bug 类型
│     ├── 自定义类型（如"故障单"）
│     └── 更多...
│
├── 项目级 Issue Type Scheme（允许创建哪些类型）
├── 项目级 Workflow Scheme（工作流→类型的映射）
├── 项目级 Screen Scheme
├── 项目级 Field Configuration Scheme
├── 项目级自定义字段（扩展全局字段）
└── 项目级看板列配置
```

### 2.2 方案映射总结

| 方案 | 说明 | LEST 表 |
|------|------|---------|
| Issue Type Scheme | 定义项目允许的 issue types | `issue_type_scheme` |
| Issue Type | 问题类型定义 | `issue_type` |
| Workflow Scheme | 工作流→类型的映射 | `workflow_scheme` |
| Workflow Definition | 工作流定义（状态+流转） | `workflow_definition` |
| Workflow Transition | 流转路径定义 | `workflow_transition` |
| Issue Type Screen Scheme | 屏幕方案→类型的映射 | `screen_scheme` |
| Screen Scheme | 操作→Screen 的映射 | `screen_scheme_detail` |
| Screen | Screen 中有哪些字段 | `screen` + `screen_field` |
| Field Configuration Scheme | 字段行为→类型的映射 | `field_config_scheme` |
| Field Configuration | 每个字段的行为（隐藏/必填/标签） | `field_config` |
| Custom Field Definition | 自定义字段元数据 | `custom_field_definition` |
| Custom Field Option | 可选项（用于 select/multi_select） | `custom_field_option` |
| Custom Field Context | 字段作用范围 | `custom_field_context` |
| Custom Field Value | 字段值（EAV） | `custom_field_value` |

## 3. 数据库设计

### 3.1 基础表

#### 3.1.1 `issue_type` — Issue Type 定义表

| 字段 | 类型 | 说明 |
|------|------|------|
| `issue_type_id` | BIGINT PK | 主键 |
| `name` | VARCHAR(64) | 显示名称：Bug / 用户故事 / 故障单 |
| `type_key` | VARCHAR(32) | 类型标识：bug / story / incident（不可重复） |
| `description` | TEXT | 类型描述 |
| `icon` | VARCHAR(64) | 图标 |
| `color` | VARCHAR(16) | 颜色 |
| `default_workflow_id` | BIGINT FK | 默认关联的工作流 |
| `default_screen_id` | BIGINT FK | 默认关联的 Screen |
| `default_field_config_id` | BIGINT FK | 默认字段配置 |
| `is_system` | TINYINT | 1=系统内置，0=自定义 |
| `sort` | INT | 排序 |
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 更新时间 |

#### 3.1.2 `workflow_definition` — 工作流定义表

| 字段 | 类型 | 说明 |
|------|------|------|
| `workflow_id` | BIGINT PK | 主键 |
| `name` | VARCHAR(128) | 工作流名称 |
| `description` | TEXT | 工作流描述 |
| `is_active` | TINYINT | 是否已激活 |
| `is_default` | TINYINT | 是否为默认工作流 |
| `created_at` | DATETIME | 创建时间 |
| `updated_at` | DATETIME | 更新时间 |

#### 3.1.3 `workflow_status` — 工作流状态表

| 字段 | 类型 | 说明 |
|------|------|------|
| `workflow_status_id` | BIGINT PK | 主键 |
| `workflow_id` | BIGINT FK | 所属工作流 |
| `status_key` | VARCHAR(32) | 状态 key：todo / in_review / done |
| `status_name` | VARCHAR(64) | 状态名称：待办 / 审核中 / 已完成 |
| `status_category` | VARCHAR(16) | 状态分类：todo / in_progress / done |
| `color` | VARCHAR(16) | 颜色 |
| `icon` | VARCHAR(32) | 图标 |
| `sort` | INT | 排序 |
| `is_initial` | TINYINT | 是否为初始状态 |

#### 3.1.4 `workflow_transition` — 流转路径表

| 字段 | 类型 | 说明 |
|------|------|------|
| `transition_id` | BIGINT PK | 主键 |
| `workflow_id` | BIGINT FK | 所属工作流 |
| `from_status_id` | BIGINT FK | 起始状态 |
| `to_status_id` | BIGINT FK | 目标状态 |
| `name` | VARCHAR(64) | 流转名称：开始 / 提交审核 / 关闭 |
| `description` | VARCHAR(255) | 流转描述 |
| `sort` | INT | 排序（决定在 UI 上的显示顺序） |

#### 3.1.5 `workflow_condition` — 流转条件表（条件触发）

| 字段 | 类型 | 说明 |
|------|------|------|
| `condition_id` | BIGINT PK | 主键 |
| `transition_id` | BIGINT FK | 所属流转 |
| `field_key` | VARCHAR(64) | 条件字段（如 `priority`、`assignee_id`） |
| `operator` | VARCHAR(16) | 操作符：eq / neq / in / not_in / is_empty / is_not_empty |
| `value` | VARCHAR(512) | 条件值 |
| `logic` | VARCHAR(8) | 逻辑组合：AND / OR |

**operator 枚举值：**

| operator | 含义 |
|----------|------|
| `eq` | 等于 |
| `neq` | 不等于 |
| `gt` / `gte` / `lt` / `lte` | 比较 |
| `in` | 在列表中 |
| `not_in` | 不在列表中 |
| `contains` | 包含 |
| `not_contains` | 不包含 |
| `is_empty` | 为空 |
| `is_not_empty` | 不为空 |
| `changed` | 值已变更 |
| `is_current_user` | 是当前用户 |
| `is_not_current_user` | 不是当前用户 |

**示例 JSON 条件配置：**

```json
{
  "conditions": [
    { "field": "assignee_id", "operator": "is_current_user", "value": null },
    { "field": "priority", "operator": "in", "value": ["p0", "p1"] }
  ],
  "logic": "AND"
}
```

#### 3.1.6 `workflow_validator` — 流转校验器表

| 字段 | 类型 | 说明 |
|------|------|------|
| `validator_id` | BIGINT PK | 主键 |
| `transition_id` | BIGINT FK | 所属流转 |

| `operator` | VARCHAR(16) | 校验操作符（同条件） |

| `error_message` | VARCHAR(255) | 校验失败提示 |
| `required` | TINYINT | 是否要求填写特定字段（如 comment） |

#### 3.1.7 `workflow_post_function` — 流转后置动作表

| 字段 | 类型 | 说明 |
|------|------|------|
| `function_id` | BIGINT PK | 主键 |
| `transition_id` | BIGINT FK | 所属流转 |
| `function_type` | VARCHAR(32) | 动作类型（见下方） |
| `function_config` | JSON | 动作配置 |
| `execute_order` | INT | 执行顺序 |
| `fail_action` | VARCHAR(8) | 失败时行为：skip / error |

**function_type 枚举：**

| function_type | 说明 | function_config 示例 |
|-------------|------|---------------------|
| `SET_FIELD` | 设置字段值 | `{"field": "status", "value": "in_progress"}` |
| `CLEAR_FIELD` | 清空字段 | `{"field": "due_date"}` |
| `COPY_FIELD` | 复制字段值 | `{"from": "title", "to": "summary"}` |
| `ADD_LABEL` | 添加标签 | `{"labels": ["reviewed"]}` |
| `REMOVE_LABEL` | 移除标签 | `{"labels": ["in_review"]}` |
| `SEND_NOTIFICATION` | 发送通知 | `{"template": "assigned", "to": "assignee"}` |
| `CREATE_SUBTASK` | 创建子任务 | `{"taskType": "task", "titleTemplate": "Review: ${title}"}` |
| `LINK_ISSUE` | 关联任务 | `{"linkType": "blocks", "issueKey": "PROJ-123"}` |
| `FIRE_WEBHOOK` | 触发 Webhook | `{"url": "https://...", "event": "status.changed"}` |
| `RUN_AUTOMATION` | 触发自动化规则 | `{"ruleId": 5}` |

#### 3.1.8 `workflow_trigger` — 工作流触发器表

| 字段 | 类型 | 说明 |
|------|------|------|

#### 3.1.9 `screen` — 屏幕配置表

| 字段 | 类型 | 说明 |
|------|------|------|
| `screen_id` | BIGINT PK | 主键 |

| `screen_type` | VARCHAR(16) | 类型：create / edit / view / transition |
| `layout` | JSON | 布局配置（Tab + 字段分组） |

| `updated_at` | DATETIME | 更新时间 |

**layout JSON 结构：**

```json
{
  "tabs": [
    {
      "tabName": "基本信息",
      "fields": [
        { "fieldKey": "title", "label": "标题", "row": 1, "col": 1 },
        { "fieldKey": "description", "label": "描述", "row": 2, "col": 1 },
        { "fieldKey": "priority", "label": "优先级", "row": 3, "col": 1 },
        { "fieldKey": "severity", "label": "严重程度", "row": 3, "col": 2 }
      ]
    },
    {
      "tabName": "人员",
      "fields": [
        { "fieldKey": "assignee_id", "label": "负责人" },
        { "fieldKey": "reporter_id", "label": "报告人" }
```

## 4. 工作流设计（完整实现）

### 4.1 工作流引擎设计

#### 4.1.1 流转校验流程

```
用户发起状态变更请求
        │
        ▼
  校验用户权限（是否有该流转的触发权限）
        │
        ▼
  加载工作流定义 + 流转路径
        │
        ▼
  校验流转条件 (workflow_condition)
  ┌─────────────────┐
  │ 条件不满足？     │───→ 返回错误 "不满足流转条件"
  └────────┬────────┘
        条件满足
        │
        ▼
  校验流转校验器 (workflow_validator)
  ┌─────────────────┐
  │ 校验不通过？     │───→ 返回错误 "校验失败: {error_message}"
  └────────┬────────┘
        校验通过
        │
        ▼
  执行前置动作（可选）
        │
        ▼
  更新 issue.status_key
        │
        ▼
  记录 issue_activity
        │
        ▼
  执行后置动作 (workflow_post_function)
        │
        ▼
  发送通知
        │
        ▼
  返回成功
```

#### 4.1.2 工作流条件示例（Jira 的 Condition）

| 场景 | 条件配置 |
|------|---------|
| 只有负责人才能完成任务 | `field: assignee_id, operator: is_current_user` |
| P0 Bug 必须测试通过才能关闭 | `field: test_status, operator: eq, value: passed` |
| 只有项目经理可以指派 | `field: transitioner_role, operator: eq, value: pm` |
| 日期未设置不能关闭 | `field: due_date, operator: is_not_empty` |

#### 4.1.3 工作流流转示例（完整的 Bug 工作流）

```
状态：新建 → 已分配 → 进行中 → 代码审核 → 测试中 → 待发布 → 已发布
         ↓
     已拒绝（需填写拒绝原因）
         ↓
     推迟（需填写推迟原因）
```

| 流转 | 条件 | 校验 | 后置动作 |
|------|------|------|---------|
| 新建→已分配 | `is_current_user: pm_or_admin` | - | 通知 assignee |
| 已分配→进行中 | - | - | 设置 start_date = now |
| 进行中→代码审核 | - | description 非空 | - |
| 代码审核→测试中 | - | 测试用例通过数 > 0 | 通知测试人员 |
| 测试中→待发布 | - | 测试通过 | - |
| 测试中→进行中 | - | - | 通知开发人员 |
| 任何→已拒绝 | - | comment 必填 | 通知 reporter |
| 任何→推迟 | - | due_date 更新 | 重新计算里程碑 |

### 6.2 工作流执行引擎

```java
@Service
public class WorkflowEngine {

    @Transactional
    public void executeTransition(Issue issue, String targetStatusKey, String comment, Long operatorId) {
        // 1. 查找流转路径
        WorkflowTransition transition = findTransition(issue, targetStatusKey);

        // 2. 校验权限（条件）
        if (!evaluateConditions(transition.getConditions(), issue, operatorId)) {
            throw new WorkflowConditionException("不满足流转条件");
        }

        // 3. 校验数据（校验器）
        if (!evaluateValidators(transition.getValidators(), issue, targetStatusKey)) {
            throw new WorkflowValidatorException(transition.getValidatorError());
        }

        // 4. 前置动作
        executePreFunctions(transition.getPreFunctions(), issue);

        // 5. 更新状态
        String oldStatus = issue.getStatusKey();
        issue.setStatusKey(targetStatusKey);
        issueService.update(issue);

        // 6. 后置动作
        executePostFunctions(transition.getPostFunctions(), issue);

        // 7. 记录活动
        activityService.record(new TransitionActivity(issue, oldStatus, targetStatusKey, comment, operatorId));

        // 8. 发送通知
        notificationService.sendTransitionNotification(issue, transition, operatorId);
    }

    // 条件引擎（支持 AND / OR 组合）
    private boolean evaluateConditions(List<WorkflowCondition> conditions, Issue issue, Long userId) {
        boolean result = conditions.isEmpty() ? true : conditions.get(0).evaluate(issue, userId);
        for (int i = 1; i < conditions.size(); i++) {
            WorkflowCondition c = conditions.get(i);
            if (c.getLogic() == Logic.AND) {
                result = result && c.evaluate(issue, userId);
            } else {
                result = result || c.evaluate(issue, userId);
            }
        }
        return result;
    }

    // 动作执行引擎
    private void executePostFunctions(List<PostFunction> functions, Issue issue) {
        for (PostFunction fn : functions) {
            fn.execute(issue);
        }
    }
}
```

```java
@Service
public class CustomFieldValueService {

    @Transactional
    public void saveFieldValues(Long issueId, Map<String, Object> values) {
        for (Map.Entry<String, Object> entry : values.entrySet()) {
            String fieldKey = entry.getKey();
            Object value = entry.getValue();
            CustomFieldDefinition field = fieldService.getByKey(fieldKey);

            CustomFieldValue cfv = valueService.findByIssueAndField(issueId, field.getFieldId());
            if (cfv == null) {
                cfv = new CustomFieldValue();
                cfv.setIssueId(issueId);
                cfv.setFieldId(field.getFieldId());
            }

            // 根据字段类型存储到对应列
            switch (field.getFieldType()) {
                case "number": cfv.setNumberValue(toNumber(value)); break;
                case "datepicker": cfv.setDateValue(toDate(value)); break;
                case "select": cfv.setOptionValue(toString(value)); break;
                default: cfv.setTextValue(toString(value));
            }
            valueService.save(cfv);
        }
    }

    public Map<String, Object> getFieldValues(Long issueId) {
        List<CustomFieldValue> values = valueService.findByIssue(issueId);
        Map<String, Object> result = new HashMap<>();
        for (CustomFieldValue v : values) {
            CustomFieldDefinition field = fieldService.getById(v.getFieldId());
            result.put(field.getFieldKey(), extractValue(v, field.getFieldType()));
        }
        return result;
    }

    // 列表页 JQL 风格查询
    public Page<Issue> searchByField(String fieldKey, String operator, Object value, PageBounds bounds) {
        CustomFieldDefinition field = fieldService.getByKey(fieldKey);
        String sql = buildFieldSql(field, operator, value);
        return jdbcTemplate.query(sql, bounds);
    }
}
```

## 9. Issue Link（任务关联）← V2.0 新增

> **Jira 对应**：Issue Links。Issue Link 是 Jira Issue Detail 页面 Sidebar 的核心功能，允许任务之间建立多种关系（blocks、relates to、duplicates 等）。

### 9.1 功能说明

任务关联用于表达任务之间的关系，例如：
- **Blocks**（阻塞）：A 阻塞 B，A 完成前 B 不能完成
- **Relates to**（关联）：A 与 B 有关联关系

- **Is duplicated by**（被重复）：A 被 B 重复
- **Cloners**（克隆来源）：A 是 B 的克隆源
- **Is cloned by**（被克隆）：A 是 B 的克隆
- **Causes**（导致）：A 导致 B 发生
- **Caused by**（被导致）：A 被 B 导致

## 13. 工作流设计增强（V2.0）

### 13.1 全局转换（Global Transition）

> **Jira 对应**：Global Transition。任何状态都可以有一个「全局」流转路径，允许在任意状态下直接转到目标状态。

**使用场景**：
- 「重新打开」：任何状态 → Open（除了已经是 Open 的）
- 「取消」：任意状态 → Cancelled
- 「挂起」：任意状态 → On Hold

**设计**：`workflow_transition` 表增加 `is_global TINYINT` 字段：
- `is_global = 1`：该流转是全局转换，不限制起始状态
- `is_global = 0`：该流转是普通流转，需要严格匹配 from_status

**校验逻辑**：
```java
// 工作流流转校验
public boolean canTransition(Issue issue, String targetStatus) {
    List<WorkflowTransition> transitions = getTransitions(issue.getStatusKey());
    // 全局转换：任何状态都可以
    for (WorkflowTransition t : transitions) {
        if (t.getToStatusKey().equals(targetStatus) && t.getIsGlobal() == 1) {
            return true;
        }
    }
    // 普通流转：必须 from_status 匹配
    for (WorkflowTransition t : transitions) {
        if (t.getToStatusKey().equals(targetStatus) && t.getFromStatusKey().equals(issue.getStatusKey())) {
            return true;
        }
    }
    return false;
}
```

### 13.2 工作流循环检测

**问题**：如果工作流配置了循环路径（如 A→B, B→A），会导致 Issue 在两个状态间无限流转。

**解决方案**：
1. **保存时检测**：发布工作流前，DFS 检测是否存在环
2. **运行时兜底**：即使有环，流转记录会防止无限循环（通过 transition_id 去重）

```java
public boolean hasCycle(WorkflowDefinition workflow) {
    Map<String, List<String>> graph = buildGraph(workflow);
    Set<String> visited = new HashSet<>();
    Set<String> recursion = new HashSet<>();
    for (String node : graph.keySet()) {
        if (dfs(node, graph, visited, recursion)) {
            return true;
        }
    }
    return false;
}

private boolean dfs(String node, Map<String, List<String>> graph,
                    Set<String> visited, Set<String> recursion) {
    if (recursion.contains(node)) return true;  // 找到环
    if (visited.contains(node)) return false;
    visited.add(node);
    recursion.add(node);
    for (String next : graph.getOrDefault(node, Collections.emptyList())) {
        if (dfs(next, graph, visited, recursion)) return true;
    }
    recursion.remove(node);
    return false;
}
```

### 13.3 工作流草稿 / 发布区分

**Jira 的 Draft/Published 机制**：
- 创建/编辑工作流时为 **草稿（Draft）** 状态
- 草稿可以反复修改
- 只有 **发布（Published）** 的工作流才会在 Issue 流转中被使用
- 发布前可预览影响范围（哪些项目/Issue Type 在使用）

**设计**：`workflow_definition` 表增加字段：

|| 字段 | 类型 | 说明 |
||------|------|------|
| `is_draft` | TINYINT | 1=草稿，0=已发布 |
| `draft_updated_at` | DATETIME | 草稿更新时间 |
| `published_at` | DATETIME | 发布时间 |

**API**：
```
# 发布草稿工作流
POST /workflow/{workflowId}/publish
响应: { "affectedProjects": 3, "affectedIssueTypes": 5 }

# 丢弃草稿
DELETE /workflow/{workflowId}/draft
```

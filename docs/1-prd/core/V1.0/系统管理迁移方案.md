# 前端 / 后端系统管理模块迁移方案

> 本文档明确 `frontend-pc/src/views/system/` 和 `backend/lest-modules/lest-system/` 中哪些需要删除、哪些需要保留，以及如何重新定位。
>
> 对应 PRD：`../1-prd/core/V1.0/认证系统与系统管理.md` 和 `../1-prd/core/V1.0/系统设置.md`

---

## 一，前端 `views/system/` 迁移

### 1.1 整体评估

```
frontend-pc/src/views/
├── system/
│   ├── user/         ← ❌ 移除（改为项目成员管理）
│   ├── role/         ← ❌ 移除（改为项目级角色）
│   ├── menu/         ← ❌ 移除（菜单固定化）
│   ├── dept/         ← ❌ 移除（敏捷平台不需要部门）
│   ├── post/         ← ❌ 移除（敏捷平台不需要岗位）
│   ├── dict/         ← ❌ 移除（用自定义字段代替）
│   ├── dict-data/    ← ❌ 移除
│   ├── config/      ← ✅ 保留（改为平台配置，admin 入口）
│   └── notice/      ← ❌ 移除（用通知系统代替）
├── monitor/
│   ├── operlog/     ← ✅ 保留（改为审计日志）
│   ├── logininfor/  ← ✅ 保留（审计日志的一部分）
│   ├── online/      ← ❌ 移除（不需要管理在线用户）
│   ├── cache/       ← ❌ 移除（运维层面）
│   ├── server/     ← ❌ 移除（运维层面）
│   └── job/         ← ❌ 移除（运维层面）
└── profile/         ← ✅ 保留（个人中心）
```

### 1.2 详细说明

#### ❌ 移除的页面

| 页面 | 移除理由 |
|------|---------|
| `system/user/` | 敏捷平台用户通过邀请/SSO 加入，不在后台录入 |
| `system/role/` | 全局角色-菜单权限体系删除，改用项目级角色 |
| `system/menu/` | 功能菜单固定，不需要管理员配置 |
| `system/dept/` | 敏捷团队不以部门为单位组织 |
| `system/post/` | 岗位是 HR 概念，敏捷平台不需要 |
| `system/dict/` + `dict-data/` | 字典管理删除，用 EAV 自定义字段代替 |
| `system/notice/` | 公告不是敏捷平台的核心需求，用通知代替 |
| `monitor/online/` | 不需要管理在线用户 |
| `monitor/cache/` | 缓存监控是运维需求 |
| `monitor/server/` | 服务器监控是运维需求 |
| `monitor/job/` | 定时任务是后端基础设施 |

#### ✅ 保留/重命名的页面

| 页面 | 保留原因 | 迁移方向 |
|------|---------|---------|
| `system/config/` | 平台级配置（邮件、安全）| 改为平台设置，入口限 platform_admin |
| `monitor/operlog/` | 审计日志（平台级操作）| 改为审计日志，限 platform_admin |
| `monitor/logininfor/` | 登录日志（审计用）| 合并到审计日志中 |
| `profile/` | 用户个人中心 | 保留，完善个人信息修改功能 |

### 1.3 新增的页面

| 页面 | 说明 | 对应 PRD |
|------|------|---------|
| `project/members/` | 项目成员管理（邀请、移除、角色变更）| 认证系统 PRD US-009~012 |

### 2.2 数据库迁移

| 操作 | 表名 | 说明 |
|------|------|------|
| 删除 | `sys_dept` | 删除部门表 |
| 删除 | `sys_post` | 删除岗位表 |
| 删除 | `sys_dict` + `sys_dict_data` | 删除字典相关表 |
| 删除 | `sys_menu` | 删除菜单配置表（或大幅精简）|
| 删除 | `sys_role_menu` | 删除角色-菜单关联表 |
| 删除 | `sys_user_role` | 删除用户-角色关联表 |
| 删除 | `sys_notice` | 删除公告表 |
| 保留 | `sys_user` | 精简字段（删除 org_id）|
| 保留 | `sys_role` | 简化（仅保留 platform_admin）|
| 新增 | `project_member` | 项目成员表 |
| 新增 | `platform_config` | 平台配置表 |
| 新增 | `audit_log` | 审计日志表（合并 operlog + logininfor）|

## 三、迁移优先级

### Phase 1：删除明显错误的功能（不影响现有业务）

1. 删除 `system/dept/` 前端页面 + 后端 Dept 相关代码
2. 删除 `system/post/` 前端页面 + 后端 Post 相关代码
3. 删除 `system/menu/` 前端页面 + 后端 Menu 相关代码
4. 删除 `system/dict/` + `dict-data/` 前端页面 + 后端 Dict 相关代码

### Phase 2：核心迁移（需要数据库迁移脚本）

1. 重写 `sys_user` 表（删除 org_id 字段）
2. 删除 `sys_role_menu` 和 `sys_user_role`，简化角色体系
3. 新增 `project_member` 表（项目成员）
4. 新增 `platform_config` 表（平台配置）
5. 新增 `audit_log` 表（审计日志）
6. 前端新增项目成员管理页面
7. 前端 `system/config/` 改造为平台设置（限 Admin）
8. 前端 `monitor/operlog/` + `logininfor/` 合并为审计日志

### Phase 3：善后

1. 删除 `system/role/` 前端页面（改用项目角色）
2. 删除 `system/user/` 中的管理员 CRUD（保留邀请制接口）
3. 删除 `system/notice/` 前端页面（用通知代替）
4. 删除 `monitor/online/` / `cache/` / `server/` / `job/`

---

## 四、迁移注意事项

1. **不要删除 `sys_user` 表中的已有用户数据**，只需删除 org_id 等无用字段
2. **保留现有的 platform_admin 用户**，确保平台管理员权限不丢失
3. **数据库迁移脚本需要回滚方案**，确保可以恢复到 V1.0
4. **前端路由需要同步更新**，删除的页面不能留孤立的路由引用
5. **API 权限注解需要清理**，删除的接口不能留未清理的 `@PreAuthorize` 注解

## 概述

安全增强模块为 LEST Platform 提供企业级安全保障，涵盖身份认证、API 访问和登录防护三个维度：

| 功能 | 说明 | 实现版本 |
|------|------|---------|
| 两步验证（2FA）| TOTP + 恢复码，防止账户被盗 | V1.0 |
| Personal Access Token | 用户级 API Token，支持细粒度权限控制 | V1.0 |
| 暴力破解防护 | 连续失败锁定 + IP 级别封禁 | V1.0 |

| 短信/邮件验证码 | 备选 2FA 方式 | V2.0 |
| 硬件 Key（FIDO2）| 高安全场景下的无密码认证 | V2.0 |

## 1. 两步验证（2FA / MFA）

### 1.1 功能说明

两因素认证（2FA）要求用户在密码之外再提供一个额外认证因素，大幅提升账户安全等级。

### 1.2 支持的验证方式

| 方式 | 说明 | 支持状态 |
|------|------|---------|
| **TOTP** | 基于时间的一次性密码（Google Authenticator / Authy / Microsoft Authenticator） | ✅ V1.0 实现 |
| **短信验证码** | 手机短信接收验证码 | ⚠️ V2.0 企业增强（需要 SMS 服务） |
| **邮件验证码** | 邮箱接收验证码 | ⚠️ V2.0 企业增强 |
| **硬件 Key** | FIDO2/WebAuthn（YubiKey 等）| ⚠️ V2.0 企业增强 |
| **恢复码** | 一次性恢复码（8位 x 10组）| ✅ V1.0 实现 |

### 1.3 用户故事

#### US-2FA-001: 启用两步验证
**作为** 用户，**我希望** 启用两步验证，**以便** 防止密码泄露导致的账户被盗。

验收标准：
- [ ] 进入「账号安全」→「两步验证」
- [ ] 选择 TOTP 方式，显示二维码
- [ ] 用身份验证器 App 扫码，显示 6 位验证码
- [ ] 输入验证码确认，完成绑定
- [ ] 生成 10 组一次性恢复码，提示用户保存
- [ ] 启用后，下次登录需要输入验证码

#### US-2FA-002: 强制启用 2FA
**作为** 平台管理员，**我希望** 强制所有用户启用两步验证，**以便** 提高平台整体安全级别。

验收标准：
- [ ] 在平台设置中开启「强制 2FA」
- [ ] 开启后，未启用 2FA 的用户首次访问时被强制引导设置
- [ ] 强制 2FA 期间可设置宽限期（7/14/30 天）

#### US-2FA-003: 信任设备
**作为** 用户，**我希望** 将常用设备标记为信任设备，**以便** 在信任设备上不重复验证。

验收标准：
- [ ] 启用 2FA 后，可选择「信任此设备 30 天」
- [ ] 信任设备在 30 天内无需重复 2FA
- [ ] 用户可在「已信任设备」列表中撤销信任

#### US-2FA-004: 使用恢复码登录
**作为** 用户，**我希望** 在无法使用 2FA 时使用恢复码登录，**以便** 避免被锁死在账户外。

验收标准：
- [ ] 登录页选择「使用恢复码」
- [ ] 输入一组恢复码，系统验证后允许登录
- [ ] 使用过的恢复码自动作废

#### US-2FA-005: 关闭两步验证
**作为** 用户，**我希望** 关闭两步验证，**以便** 在不需要时简化登录流程。

验收标准：
- [ ] 进入「账号安全」→「两步验证」
- [ ] 输入当前密码 + 当前 2FA 验证码
- [ ] 验证通过后关闭 2FA
- [ ] 平台管理员可强制关闭（用于处理用户账户问题）

### 1.4 数据库设计

```sql
-- 用户 2FA 配置
CREATE TABLE user_mfa_config (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id         BIGINT NOT NULL UNIQUE,
    mfa_type        VARCHAR(16) NOT NULL COMMENT 'TOTP/SMS/EMAIL/FIDO2',
    secret_key      VARCHAR(255) COMMENT 'TOTP Secret（加密存储）',
    phone           VARCHAR(32) COMMENT '手机号（SMS 用）',
    is_enabled      BOOLEAN DEFAULT FALSE,
    force_enabled   BOOLEAN DEFAULT FALSE COMMENT '平台强制启用',
    enabled_at      DATETIME,
    last_verified   DATETIME,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES sys_user(user_id)
);

-- 信任设备
CREATE TABLE trusted_device (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id         BIGINT NOT NULL,
    device_token    VARCHAR(256) NOT NULL COMMENT '设备指纹 token',
    device_name     VARCHAR(128) COMMENT '设备名称',
    expires_at      DATETIME NOT NULL COMMENT '信任过期时间',
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES sys_user(user_id)
);

-- 恢复码
CREATE TABLE mfa_recovery_code (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id         BIGINT NOT NULL,
    code_hash       VARCHAR(255) NOT NULL COMMENT 'BCrypt 加密的恢复码',
    is_used         BOOLEAN DEFAULT FALSE,
    used_at         DATETIME,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES sys_user(user_id)
);

-- 登录失败记录（用于暴力破解检测）
CREATE TABLE login_attempt (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id         BIGINT,
    username        VARCHAR(64) NOT NULL,
    ip             VARCHAR(64) NOT NULL,
    user_agent     VARCHAR(512),
    success        BOOLEAN NOT NULL,
    mfa_passed     BOOLEAN COMMENT '2FA 是否通过',
    failed_reason  VARCHAR(64),
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_time (user_id, created_at),
    INDEX idx_ip_time (ip, created_at)
);
```

## 2. API Token / Personal Access Token（PAT）

### 2.1 功能说明

Personal Access Token（PAT）是用户个人的 API 访问令牌，用于替代密码进行 API 认证，支持设置过期时间和权限范围。

### 2.2 与 API Key 的区别

| 维度 | API Key | Personal Access Token（PAT） |
|------|---------|------------------------------|
| **生成方式** | 管理员生成，全局共享 | 用户自己生成，个人使用 |
| **粒度** | 全局或项目级 | 按 Token 设置权限范围 |
| **过期时间** | 永久（可手动撤销）| 可设置过期时间 |
| **可见性** | 可共享给他人 | 仅创建者可见（Token 只显示一次）|
| **使用场景** | CI/CD、集成服务账号 | 个人开发脚本、API 测试 |

### 2.3 用户故事

#### US-PAT-001: 创建 Personal Access Token
**作为** 用户，**我希望** 创建自己的 API Token，**以便** 使用 API 访问我的数据。

验收标准：
- [ ] 进入「个人设置」→「开发者选项」→「Personal Access Token」
- [ ] 输入 Token 名称（如「CI 脚本」「个人测试」）
- [ ] 设置权限范围（任务:读/写、代码:读/写、用户:读 等）
- [ ] 设置过期时间（30天/90天/180天/永不过期）
- [ ] 点击创建，系统显示 Token（**只显示一次**，刷新后不再显示）
- [ ] 提供复制到剪贴板按钮

#### US-PAT-002: 管理 Token
**作为** 用户，**我希望** 查看和撤销自己的 Token，**以便** 管理 API 访问权限。

验收标准：
- [ ] Token 列表显示：名称 / 权限范围 / 过期时间 / 最近使用 / 创建时间
- [ ] 显示每个 Token 的最近使用时间
- [ ] 可随时撤销 Token（Revoke）
- [ ] 撤销后 Token 立即失效
- [ ] Token 过期后自动失效

#### US-PAT-003: Token 权限控制
**作为** 平台管理员，**我希望** 限制普通用户创建 Token 的权限范围，**以便** 防止 Token 权限过大。

验收标准：
- [ ] 在平台设置中配置：哪些权限范围允许普通用户创建
- [ ] 管理员可完全禁用 PAT 功能
- [ ] 管理员可查看所有用户的 Token 列表（隐藏 Token 值）

### 2.4 数据库设计

```sql
-- Personal Access Token
CREATE TABLE personal_access_token (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id         BIGINT NOT NULL,
    token_name      VARCHAR(128) NOT NULL,
    token_prefix    VARCHAR(8) NOT NULL COMMENT 'Token 前 8 位（用于列表展示）',
    token_hash      VARCHAR(255) NOT NULL COMMENT 'BCrypt 哈希存储',
    scopes          JSON NOT NULL COMMENT '["task:read","task:write","project:read"]',
    expires_at      DATETIME COMMENT 'NULL 表示永不过期',
    last_used_at    DATETIME,
    last_used_ip    VARCHAR(64),
    last_used_ua   VARCHAR(512),
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    revoked_at      DATETIME,
    FOREIGN KEY (user_id) REFERENCES sys_user(user_id)
);

-- Token 使用记录（可选，用于审计）
CREATE TABLE token_usage_log (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    token_id        BIGINT NOT NULL,
    endpoint        VARCHAR(256) NOT NULL,
    method          VARCHAR(8),
    ip              VARCHAR(64),
    status_code     INT,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (token_id) REFERENCES personal_access_token(id)
);
```

### 2.5 API Token 使用方式

```bash
# 使用 PAT 认证 API
curl -H "Authorization: Bearer pat_xxxxxxxxxxxxxxxxxxxx" \
     https://api.lest.dev/v1/issues

# 权限范围示例
# task:read     - 读取任务
# task:write    - 创建/编辑/删除任务
# project:read  - 读取项目信息
# project:write - 创建/编辑项目
# user:read     - 读取用户信息
# admin:write   - 管理功能（仅管理员）
```

## 3. 登录安全增强

### 3.1 暴力破解防护

在现有登录锁定基础上，增加更细粒度的防护：

| 防护项 | 说明 |
|--------|------|
| **IP 级限流** | 同一 IP 5 分钟内最多 20 次登录尝试 |
| **账号锁定** | 连续 5 次密码错误，锁定 15 分钟 |
| **CAPTCHA 触发** | 连续 3 次登录失败，显示图形验证码 |
| **异常登录提醒** | 检测到异地登录，自动发送邮件提醒 |
| **登录失败日志** | 记录所有失败尝试（IP/时间/原因）|

### 3.2 可信网络

| 功能 | 说明 |
|------|------|
| **IP 白名单** | 管理员可配置可信 IP 列表，白名单内 IP 免 2FA |
| **IP 黑名单** | 管理员可配置禁用 IP 列表 |

### 3.3 数据库设计（扩展）

```sql
-- 可信网络配置
CREATE TABLE network_trust (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    trust_type      VARCHAR(8) NOT NULL COMMENT 'WHITE/BLACK',
    ip_pattern      VARCHAR(64) NOT NULL COMMENT '支持 CIDR 格式 192.168.1.0/24',
    description     VARCHAR(256),
    created_by      BIGINT NOT NULL,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (created_by) REFERENCES sys_user(user_id)
);
```

## 5. 错误码

| 模块 | 错误码范围 | 说明 |
|------|-----------|------|
| 2FA | 21000-21099 | 两步验证模块 |
| PAT | 22000-22099 | Personal Access Token 模块 |
| 登录安全 | 23000-23099 | 暴力破解防护 / 可信网络模块 |

### 5.1 2FA 错误码

| 错误码 | 说明 | HTTP 状态 |
|--------|------|---------|
| 21001 | 2FA 未启用，无法验证 | 400 |
| 21002 | TOTP 验证码错误 | 401 |
| 21003 | TOTP 验证码已使用（重放攻击防护）| 400 |
| 21004 | TOTP 验证码过期（超过 30 秒窗口）| 400 |
| 21005 | 恢复码已被使用 | 400 |
| 21006 | 恢复码无效 | 400 |
| 21007 | 2FA 强制启用中，不允许跳过 | 403 |

### 5.2 PAT 错误码

| 错误码 | 说明 | HTTP 状态 |
|--------|------|---------|
| 22001 | Token 不存在或已撤销 | 404 |
| 22002 | Token 已过期 | 401 |
| 22003 | Token 权限不足（权限范围不包含请求的操作）| 403 |
| 22004 | Token 被禁用 | 403 |
| 22005 | 令牌格式无效 | 400 |

### 5.3 登录安全错误码

| 错误码 | 说明 | HTTP 状态 |
|--------|------|---------|
| 23001 | 账户已被锁定（暴力破解触发）| 423 |
| 23002 | IP 已被封禁 | 403 |
| 23003 | 登录失败次数过多，请稍后再试 | 429 |
| 23004 | 可信设备未在白名单中 | 403 |

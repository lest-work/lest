## 1. 核心设计理念：LEST 不是 OA，不是 ERP

### 1.1 传统管理系统 vs 敏捷平台的根本区别

| 维度 | 传统管理系统（OA/ERP） | LEST 敏捷平台（Jira 思路）|
|------|------|------|
| **用户来源** | 管理员在后台录入员工 | 用户自行注册或通过 SSO/邀请链接加入 |
| **组织结构** | 部门-岗位-员工三级树 | 无固定组织层级；通过"项目成员"灵活协作 |
| **权限模型** | 全局菜单树（菜单-按钮级别）| 项目级角色（Admin/Dev/Viewer），菜单按功能动态展示 |
| **管理员职责** | 管理员工账号、分配菜单权限 | 管理项目成员、指派项目角色；平台级仅管安全配置 |
| **典型用户** | HR、财务、行政 | 开发者、测试、产品、项目经理 |

### 1.2 哪些要删除（不再做）

> **删除理由**：这些是 OA/ERP 的标配，但不属于敏捷管理平台的范畴。

| 删除项 | 删除理由 | 替代方案 |
|--------|---------|---------|
| 部门管理（机构树）| 敏捷团队按项目组织，不需要固定部门层级 | 通过项目成员列表管理团队 |
| 岗位管理 | "岗位"是 HR 概念，敏捷平台不需要 | 通过项目角色（Dev/QA/PM）代替 |
| 字典管理 | "字典"是管理数据枚举的方式，敏捷平台用自定义字段代替 | EAV 自定义字段（V2.0）|
| 菜单管理 | 菜单是系统功能，不是管理员可配置的内容 | 功能菜单固定，按用户角色自动展示 |
| 全局用户列表（管理员录入式）| 用户应通过邀请或 SSO 加入，而非管理员创建 | 用户邀请 + SSO（V3.0 lest-sso/lest-ldap）|
| 全局角色-菜单权限体系 | 菜单权限是传统系统做法，敏捷平台按项目授权 | 项目级 Permission Scheme |

### 1.3 哪些要保留或新增

| 保留/新增 | 说明 | 对应 Jira |
|---------|------|---------|
| JWT 认证 | 登录、登出、Token 刷新 | Atlassian Account |
| 平台管理员 | 平台级配置（仅超级管理员）| Jira System Admin |
| 审计日志 | 平台级操作记录 | Jira Audit Log |
| 项目成员管理 | 邀请/移除项目成员 | Jira Project Roles |
| 通知渠道配置 | 邮件/IM 渠道配置 | Jira Email / Notification |
| 个人资料 | 用户修改昵称、头像、密码 | 用户个人中心 |

## 2. 权限模型：项目级 Permission Scheme

### 2.1 权限的三个层次

LEST 的权限分为三层，不同于传统系统的全局菜单权限：

```
第一层：平台级（Platform Admin Only）
  └── 平台管理员（超级管理员）才能访问
      - 系统设置（邮件、安全、存储）
      - 审计日志查看
      - 插件管理

第二层：项目级（Project Members）
  └── 所有项目成员都有的基础权限
      - 查看自己有权访问的项目
      - 接收通知
  └── 项目角色决定具体权限
      - 项目管理员（Admin）：管理项目设置、成员、配置
      - 开发者（Dev）：创建/编辑任务、提交代码
      - 查看者（Viewer）：只读访问

第三层：任务级
  └── 任务负责人或被授权者
      - 编辑具体任务
      - 评论、附件
```

### 2.2 项目内置角色（Jira 风格）

| 角色 | Jira 对应 | 权限描述 |
|------|---------|---------|
| **Admin**（项目管理员） | Project Administrator | 完全控制项目：管理成员、设置、配置 |
| **Member**（成员） | Project Member | 常规操作：创建/编辑任务、评论、代码关联 |
| **Viewer**（查看者） | Viewer | 只读：查看任务、看板、报表 |

> **注意**：这里 Member 是所有实际干活的成员（开发者、测试、产品），不是"普通员工"。

### 2.3 权限继承关系

```
平台管理员（Platform Admin）
    │
    ├── 可以访问平台级设置（系统配置、审计日志）
    ├── 可以管理所有项目（作为任何项目的 Admin）
    │
    └── 可以创建新项目（自动成为该项目的 Admin）

项目管理员（Admin）
    ├── 管理项目成员（添加/移除/变更角色）
    ├── 管理项目设置（名称、描述、迭代）
    ├── 管理项目级 Scheme（Issue Type Scheme 等）
    └── 无权访问平台级设置

项目成员（Member）
    ├── 创建/编辑/删除自己的任务
    ├── 评论、附件、工时记录
    └── 无权管理项目和平台

查看者（Viewer）
    ├── 只读访问项目内容
    └── 无权修改任何内容
```

## 3. 用户故事

### 3.1 认证模块（保留并完善）

#### US-001: 用户登录
**作为** 普通用户，**我希望** 通过用户名和密码登录平台，**以便** 访问平台功能。

验收标准：
- [ ] 用户输入正确的用户名和密码，系统验证通过后返回 Token
- [ ] 错误提示不区分"用户名不存在"还是"密码错误"（防止枚举攻击）
- [ ] 支持图形验证码防暴力破解
- [ ] 登录成功后返回 Access Token（15分钟）和 Refresh Token（7天）

#### US-002: 刷新 Token
**作为** 登录用户，**我希望** 在 Access Token 过期前无感刷新，**以便** 持续使用平台不中断。

验收标准：
- [ ] 客户端自动用 Refresh Token 换取新 Access Token
- [ ] Refresh Token 过期后跳转登录页

#### US-003: 用户登出
**作为** 登录用户，**我希望** 退出登录，**以便** 清除本地会话。

验收标准：
- [ ] 登出后 Token 立即失效
- [ ] 客户端清除所有缓存数据

#### US-004: 修改密码
**作为** 登录用户，**我希望** 修改自己的登录密码，**以便** 提高账户安全性。

验收标准：
- [ ] 需验证当前密码
- [ ] 新密码 8-32 位，必须包含字母和数字
- [ ] 修改后所有 Refresh Token 自动失效

#### US-005: 个人资料管理
**作为** 登录用户，**我希望** 修改自己的资料，**以便** 保持个人信息准确。

验收标准：
- [ ] 可修改：昵称、头像、邮箱
- [ ] 不可修改：用户名（用户名即账号标识）

### 3.2 平台管理模块（精简，只做平台级配置）

#### US-006: 平台管理员访问
**作为** 平台管理员，**我希望** 在专用入口访问平台管理后台，**以便** 配置平台级参数。

> **设计说明**：平台管理不等于"系统管理"菜单。平台管理员在项目列表页的「管理」入口进入平台设置，而非左侧菜单。

验收标准：
- [ ] 平台设置入口仅对 `platform_admin` 角色可见
- [ ] 平台设置包含：邮件配置、通知渠道、安全设置、审计日志、插件管理

#### US-007: 配置邮件服务
**作为** 平台管理员，**我希望** 配置邮件发送服务，**以便** 发送通知邮件。

验收标准：
- [ ] 配置 SMTP 服务器（地址、端口、账号密码）
- [ ] 支持 TLS/SSL
- [ ] 可发送测试邮件验证配置

#### US-008: 查看审计日志
**作为** 平台管理员，**我希望** 查看平台级操作日志，**以便** 审计安全事件。

验收标准：
- [ ] 记录：用户登录/登出、密码修改、项目创建/删除、人员变更等平台级操作
- [ ] 支持按用户、时间范围、操作类型筛选
- [ ] 审计日志不可删除，仅可查询
- [ ] 日志保留 90 天

### 3.3 项目成员管理模块（新增，按 Jira 思路）

#### US-009: 邀请成员加入项目
**作为** 项目管理员，**我希望** 邀请成员加入项目，**以便** 让团队成员协作。

> **设计说明**：Jira 的做法不是"在用户列表里挑"，而是直接输入邮箱地址邀请。LEST V1.0 支持输入邮箱或用户名邀请；V3.0 通过 SSO 插件支持企业账号直接导入。

验收标准：
- [ ] 输入目标用户的邮箱或用户名，发起邀请
- [ ] 被邀请用户自动出现在项目成员列表中（待接受状态）
- [ ] 若被邀请用户不存在，系统创建账号并发送激活邮件
- [ ] 邀请时可指定角色（Admin / Member / Viewer）
- [ ] 支持批量发送邀请

#### US-010: 查看项目成员
**作为** 项目管理员或成员，**我希望** 查看项目成员列表，**以便** 了解团队构成。

验收标准：
- [ ] 成员列表显示：用户名、昵称、角色、加入时间、最近活跃
- [ ] 按角色筛选成员

#### US-011: 变更成员角色
**作为** 项目管理员，**我希望** 变更成员的权限角色，**以便** 调整团队权限。

验收标准：
- [ ] 可将成员从 Viewer 升为 Member，从 Member 升为 Admin
- [ ] 不可将唯一 Admin 降级（需先指定其他 Admin）
- [ ] 角色变更即时生效

#### US-012: 移除项目成员
**作为** 项目管理员，**我希望** 移除不需要的成员，**以便** 保持项目访问安全。

验收标准：
- [ ] 移除后，该成员无法再访问项目
- [ ] 移除不影响该成员在其他项目中的身份
- [ ] 移除操作记录到审计日志

### 3.4 SSO 与企业集成（V3.0 插件，架构预埋）

#### US-013: 通过 SSO 登录（V3.0 lest-sso）
**作为** 企业用户，**我希望** 使用企业账号直接登录 LEST，**以便** 无需单独注册。

> **说明**：此功能通过 `lest-sso` 插件实现（V3.0），Core 需预留对接接口。详见 `../../plugins/V3.0/lest-sso.md`。

#### US-014: LDAP 用户同步（V3.0 lest-ldap）
**作为** 平台管理员，**我希望** 从企业 LDAP 导入用户，**以便** 统一管理企业身份。

> **说明**：此功能通过 `lest-ldap` 插件实现（V3.0）。详见 `../../plugins/V3.0/lest-ldap.md`。

## 8. 与 V1.0 PRD 的差异说明

| 差异项 | V1.0（错误做法）| V2.0（Jira 思路）|
|--------|--------------|--------------|
| 用户来源 | 管理员在后台创建用户 | 用户通过邀请或 SSO 加入 |
| 组织结构 | 部门-岗位-员工三级体系 | 无固定部门，通过项目成员管理 |
| 权限模型 | 全局 SysRole → SysMenu 树 | 项目级角色（Admin/Member/Viewer）|
| 菜单权限 | 管理员配置菜单树 + 按钮权限 | 功能菜单按角色自动展示 |
| 字典管理 | 后台 CRUD 字典数据 | 用 EAV 自定义字段代替 |
| 审计日志 | 记录所有操作（包括任务操作）| 只记录平台级操作 |
| 用户导入 | Excel 批量导入 | V3.0 通过 SSO/LDAP 插件实现 |
| 菜单管理 | 独立的菜单 CRUD 页面 | 删除，功能菜单固定 |

## 1. 概述

### 1.1 V2.0 遗留的功能缺口

V2.0 完成了 Scheme 架构体系（EAV 自定义字段、Screen/Scheme、工作流引擎），但以下核心 Jira 功能仅有数据结构设计，缺少完整的用户故事和 API 设计：

| 功能 | V2.0 状态 | V3.0 补全目标 |
|------|----------|-------------|
| Issue Link | 有表结构，缺用户故事 | 完整 8 种链接类型 |
| Attachment | 缺失 | 完整附件体系 |
| Time Tracking | 缺失 | 预估/剩余/工时记录 |
| Sub-task | 基本父子关系 | 完整子任务 Issue Type |
| Automation 执行 | 有 trigger 数据结构 | 执行引擎 + 日志 |
| 回收站 | 缺失 | 软删除 + 恢复 |

### 1.2 V3.0 补全策略

V3.0 **不改变 V2.0 的 Scheme 架构**，而是在其基础上补充缺失的用户故事和 API 设计，确保每个 Jira 核心功能都有对应的实现。

## 2. Issue Link（任务关联）

### 2.1 功能说明

Jira Issue Link 允许用户在一个 Issue 上关联另一个 Issue，形成需求网络。每个链接有方向性和链接类型。

### 2.2 链接类型（8 种）

| 链接类型 key | 名称 | 方向 | 说明 |
|-------------|------|------|------|
| `blocks` | 阻塞 | 单向 | 当前任务阻塞目标任务 |
| `is_blocked_by` | 被阻塞 | 单向 | 当前任务被目标任务阻塞 |
| `duplicates` | 重复 | 单向 | 当前任务与目标任务重复 |
| `is_duplicated_by` | 被重复 | 单向 | 当前任务被目标任务重复 |
| `relates_to` | 相关 | 双向 | 两个任务相关 |
| `causes` | 导致 | 单向 | 当前任务导致目标任务 |
| `is_caused_by` | 被导致 | 单向 | 当前任务被目标任务导致 |
| `clones` | 克隆 | 单向 | 当前任务克隆自目标任务 |

### 2.3 用户故事

#### US-ISSUELINK-001: 创建任务链接
**作为** 任务负责人，**我希望** 将当前任务与其他任务关联，**以便** 表达任务之间的关系。

验收标准：
- [ ] 在任务详情页点击"添加关联"按钮
- [ ] 选择链接类型（blocks / relates_to / duplicates 等）
- [ ] 搜索并选择目标任务（支持 JQL 搜索）
- [ ] 自动发送通知给被关联任务的负责人
- [ ] 链接记录到任务操作历史

#### US-ISSUELINK-002: 查看任务链接
**作为** 团队成员，**我希望** 查看一个任务的所有链接，**以便** 了解任务之间的关系网络。

验收标准：
- [ ] 在任务详情页"关联"Tab 中显示所有链接
- [ ] 按链接类型分组显示（正向/反向）
- [ ] 点击链接跳转到目标任务
- [ ] 显示链接的创建人和创建时间
- [ ] 支持筛选特定类型的链接

#### US-ISSUELINK-003: 批量管理链接
**作为** 项目经理，**我希望** 批量创建或删除任务链接，**以便** 快速建立任务网络。

验收标准：
- [ ] 支持 CSV 导入批量创建链接
- [ ] 支持批量删除选中的链接
- [ ] 批量操作记录到审计日志

### 2.4 数据库设计

```sql
-- issue_link_type（链接类型）
CREATE TABLE issue_link_type (
  link_type_id   BIGINT PK AUTO_INCREMENT,
  link_key       VARCHAR(32) NOT NULL UNIQUE,  -- blocks / relates_to
  name           VARCHAR(64) NOT NULL,           -- 阻塞 / 相关
  inward         VARCHAR(64),                     -- 反向描述：被阻塞
  outward        VARCHAR(64),                     -- 正向描述：阻塞
  is_system      TINYINT DEFAULT 0,              -- 1=内置不可删除
  is_active      TINYINT DEFAULT 1,
  created_at     DATETIME
);

-- issue_link（任务链接）
CREATE TABLE issue_link (
  link_id        BIGINT PK AUTO_INCREMENT,
  source_issue_id BIGINT NOT NULL,               -- 源任务
  target_issue_id BIGINT NOT NULL,               -- 目标任务
  link_type_id   BIGINT NOT NULL,                -- 链接类型
  created_by     BIGINT NOT NULL,
  created_at     DATETIME,
  FOREIGN KEY (source_issue_id) REFERENCES issue(id),
  FOREIGN KEY (target_issue_id) REFERENCES issue(id),
  FOREIGN KEY (link_type_id) REFERENCES issue_link_type(link_type_id)
);
```

## 3. Attachment（附件管理）

### 3.1 功能说明

Jira 附件允许用户上传文件到任务详情页，支持图片预览、文档在线查看、版本管理。

### 3.2 用户故事

#### US-ATTACH-001: 上传附件
**作为** 任务负责人，**我希望** 上传附件到任务，**以便** 提供相关文档和截图。

验收标准：
- [ ] 在任务详情页"附件"Tab 上传文件
- [ ] 支持拖拽上传和点击选择
- [ ] 支持多文件同时上传
- [ ] 显示上传进度条
- [ ] 限制单个文件大小（如 100MB）和总附件大小（如 500MB）
- [ ] 支持的文件类型：图片、PDF、Word、Excel、Zip 等
- [ ] 自动生成缩略图（图片类型）
- [ ] 上传后记录到操作历史

#### US-ATTACH-002: 预览和下载附件
**作为** 团队成员，**我希望** 预览和下载任务附件，**以便** 查看相关文档。

验收标准：
- [ ] 图片类型在线预览（缩略图点击放大）
- [ ] PDF 和文档支持在线预览
- [ ] 支持下载到本地
- [ ] 显示文件大小、上传人、上传时间

#### US-ATTACH-003: 删除附件
**作为** 任务负责人或管理员，**我希望** 删除任务附件，**以便** 清理不需要的文件。

验收标准：
- [ ] 删除前需确认
- [ ] 删除操作记录到操作历史

#### US-ATTACH-004: 附件版本管理
**作为** 任务负责人，**我希望** 为同一文件上传新版本，**以便** 更新文档而不丢失历史版本。

验收标准：
- [ ] 同一文件名再次上传自动创建新版本

### 3.3 数据库设计

```sql
-- attachment（附件表）
CREATE TABLE attachment (
  attachment_id   BIGINT PK AUTO_INCREMENT,
  issue_id       BIGINT NOT NULL,
  file_name      VARCHAR(255) NOT NULL,
  file_size      BIGINT NOT NULL,                 -- 字节
  file_path      VARCHAR(512) NOT NULL,           -- 存储路径
  file_type      VARCHAR(64) NOT NULL,            -- MIME type
  thumbnail_path VARCHAR(512),                     -- 缩略图路径
  uploaded_by    BIGINT NOT NULL,
  uploaded_at    DATETIME,
  version       INT DEFAULT 1,                   -- 版本号
  parent_id      BIGINT,                          -- 上一个版本的 attachment_id
  is_deleted    TINYINT DEFAULT 0,
  deleted_at    DATETIME,
  deleted_by    BIGINT,
  FOREIGN KEY (issue_id) REFERENCES issue(id)
);
```

## 4. Time Tracking（时间追踪）

### 4.1 功能说明

Jira Time Tracking 包含三个核心概念：预估工时（Estimate）、剩余工时（Remaining Estimate）、已记录工时（Time Spent）。

### 4.2 用户故事

#### US-TIME-001: 设置预估工时
**作为** 任务负责人，**我希望** 设置任务的预估工时，**以便** 评估任务工作量。

验收标准：
- [ ] 在任务详情页设置预估工时（格式：1w 2d 3h 4m）
- [ ] 支持快速选择预设值（1h / 2h / 4h / 8h / 1d / 3d）
- [ ] 预估工时显示在任务卡片和列表中

#### US-TIME-002: 记录已工时
**作为** 任务负责人，**我希望** 记录在任务上花费的时间，**以便** 追踪实际工作量。

验收标准：
- [ ] 在任务详情页记录工时（工时 + 日期）
- [ ] 支持添加工时说明
- [ ] 记录后自动更新剩余工时 = 预估工时 - 已记录工时
- [ ] 剩余工时为负时给出警告
- [ ] 工时记录支持编辑和删除

#### US-TIME-003: 进度可视化
**作为** 项目经理，**我希望** 查看任务的工时进度，**以便** 了解任务完成情况。

验收标准：
- [ ] 任务卡片显示进度条（已记录/预估）
- [ ] 看板列显示该列任务的总工时
- [ ] 任务列表显示工时列

### 4.3 数据库设计

```sql
-- 在 issue 表中增加时间追踪字段（V3.0）
ALTER TABLE issue ADD COLUMN estimate_seconds BIGINT;         -- 预估工时（秒）
ALTER TABLE issue ADD COLUMN remaining_seconds BIGINT;        -- 剩余工时（秒）
ALTER TABLE issue ADD COLUMN time_spent_seconds BIGINT DEFAULT 0;  -- 已记录工时（秒）

-- worklog（工时记录）
CREATE TABLE worklog (
  worklog_id     BIGINT PK AUTO_INCREMENT,
  issue_id       BIGINT NOT NULL,
  user_id        BIGINT NOT NULL,
  started_at     DATETIME NOT NULL,                 -- 工作开始时间
  duration_seconds BIGINT NOT NULL,                  -- 工时时长（秒）
  description    VARCHAR(512),                       -- 工时说明
  created_at     DATETIME,
  updated_at     DATETIME,
  is_deleted     TINYINT DEFAULT 0,
  FOREIGN KEY (issue_id) REFERENCES issue(id)
);
```

## 5. Sub-task（子任务增强）

### 5.1 功能说明

Jira Sub-task 是专门的任务类型，用于将父任务拆分为更小的工作单元。

### 5.2 用户故事

#### US-SUBTASK-001: 创建子任务
**作为** 任务负责人，**我希望** 将一个任务拆分为多个子任务，**以便** 细分工作项。

验收标准：
- [ ] 在父任务详情页点击"添加子任务"
- [ ] 子任务继承父任务的项目、迭代
- [ ] 子任务完成后，父任务可自动感知
- [ ] 子任务可有自己的工作流（可与父任务不同）
- [ ] 子任务可单独分配给不同的人
- [ ] 父任务显示所有子任务的进度汇总

#### US-SUBTASK-002: 子任务 Issue Type
**作为** 项目管理员，**我希望** 为项目配置子任务的 Issue Type，**以便** 限制子任务的类型。

验收标准：
- [ ] 在 Issue Type Scheme 中指定子任务类型（如 Task）
- [ ] 每个父 Issue Type 关联一个子 Issue Type
- [ ] 子任务类型只能作为子任务创建，不能独立创建

### 5.3 数据库设计

```sql
-- 在 issue 表中增加父子关系字段
ALTER TABLE issue ADD COLUMN parent_id BIGINT;              -- 父任务 ID
ALTER TABLE issue ADD COLUMN is_subtask TINYINT DEFAULT 0; -- 是否为子任务
ALTER TABLE issue ADD COLUMN root_id BIGINT;                -- 根任务 ID（用于快速查询整棵任务树）

-- index 优化
CREATE INDEX idx_issue_parent ON issue(parent_id);
CREATE INDEX idx_issue_root ON issue(root_id);
```

## 6. Automation 执行引擎（补全 V2.0）

### 6.1 功能说明

V2.0 已设计 Automation Rule 的数据结构（trigger/condition/action），V3.0 需要实现执行引擎。

### 6.2 执行引擎设计

```
触发事件发生
    │
    ▼
规则引擎扫描所有启用的规则
    │
    ▼
匹配 trigger_type（issue_created / field_changed / scheduled）
    │
    ▼
评估 conditions（AND/OR 条件树）
    │
    ├── 条件不满足 → 跳过
    │
    ▼
执行 actions（按 execute_order）
    │
    ├── SET_FIELD / SEND_NOTIFICATION / CREATE_SUBTASK 等
    │
    ▼
记录执行日志（automation_execution_log）
    │
    ▼
出错则按 fail_action 处理（skip / error）
```

### 6.3 用户故事

#### US-AUTO-001: 创建自动化规则
**作为** 项目管理员，**我希望** 创建自动化规则，**以便** 减少重复性操作。

验收标准：
- [ ] 从预设模板创建规则（"任务创建时自动分配"等）
- [ ] 手动配置 trigger / conditions / actions
- [ ] 规则预览：显示满足条件时会影响的 Issue 列表
- [ ] 规则可启用/禁用

#### US-AUTO-002: 查看规则执行日志
**作为** 管理员，**我希望** 查看自动化规则的执行日志，**以便** 排查问题和审计。

验收标准：
- [ ] 显示每个规则的执行历史
- [ ] 日志包含：触发时间、影响的问题、执行的动作、是否成功
- [ ] 失败的执行显示错误原因
- [ ] 支持按规则/按问题/按时间筛选

### 6.4 数据库设计

```sql
-- automation_rule（自动化规则）
CREATE TABLE automation_rule (
  rule_id        BIGINT PK AUTO_INCREMENT,
  project_id     BIGINT,                           -- NULL=全局规则
  name           VARCHAR(128) NOT NULL,
  description    TEXT,
  trigger_type   VARCHAR(32) NOT NULL,              -- issue_created / field_changed / scheduled
  trigger_config JSON,
  conditions     JSON,                              -- 条件树
  actions       JSON,                              -- 动作列表
  is_enabled    TINYINT DEFAULT 1,
  created_by     BIGINT,
  created_at     DATETIME,
  updated_at     DATETIME
);

-- automation_execution_log（执行日志）
CREATE TABLE automation_execution_log (
  log_id         BIGINT PK AUTO_INCREMENT,
  rule_id        BIGINT NOT NULL,
  trigger_event  VARCHAR(64) NOT NULL,              -- 触发事件描述
  issue_id       BIGINT,                            -- 关联的问题
  executed_by    VARCHAR(32),                       -- system / user_id
  actions_executed JSON,                            -- 执行的动作
  status         VARCHAR(16),                       -- success / failed / skipped
  error_message  TEXT,
  executed_at    DATETIME,
  FOREIGN KEY (rule_id) REFERENCES automation_rule(rule_id)
);
```

#### US-TRASH-001: 软删除

#### US-TRASH-002: 查看回收站
**作为** 项目管理员，**我希望** 查看回收站中的已删除任务，**以便** 决定是否恢复。

验收标准：

#### US-TRASH-003: 恢复任务
**作为** 项目管理员，**我希望** 恢复误删的任务，**以便** 还原工作内容。

验收标准：
- [ ] 一键恢复任务及其子任务
- [ ] 恢复后任务回到原项目
- [ ] 恢复操作记录到操作历史

#### US-TRASH-004: 彻底删除
**作为** 平台管理员，**我希望** 彻底删除回收站中的任务，**以便** 清理无用数据。

验收标准：
- [ ] 彻底删除前需二次确认
- [ ] 彻底删除后不可恢复

- [ ] 记录到审计日志

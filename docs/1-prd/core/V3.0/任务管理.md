## 1. 概述

### 1.1 V2.0 遗留的功能缺口

V2.0 完成了 Scheme 架构体系（EAV 自定义字段、Screen/Scheme、工作流引擎），但以下核心 Jira 功能仅有数据结构设计，缺少完整的用户故事和 API 设计：

| 功能 | V2.0 状态 | V3.0 补全目标 |
|------|----------|-------------|
| Issue Link | 有表结构，缺用户故事 | 完整 8 种链接类型 |
| Attachment | 缺失 | 完整附件体系 |
| Time Tracking | 缺失 | 预估/剩余/工时记录 |
| Sub-task | 基本父子关系 | 完整子任务 Issue Type |
| Automation 执行 | 有 trigger 数据结构 | 执行引擎 + 日志 |
| 回收站 | 缺失 | 软删除 + 恢复 |
| Vote / Share | 缺失 | 完整投票 + 分享体系 |
| 批量操作 | 缺失 | 批量修改/移动/删除 |
| 键盘快捷键 | 缺失 | 完整快捷键体系 |
| Automation Rule Templates | 缺失 | 规则模板库 |

此外，以下能力在 V2.0 PRD 中**已覆盖设计**，但若未在 V2.0 完整落地，则统一纳入 V3.0 作为 **Jira 100% 功能收口项**实现并验收：

- 批量操作（Batch Operations）
- 投票/分享（Vote / Share）
- 键盘快捷键（Shortcuts）
- 自动化规则模板库（Automation Rule Templates）

### 1.2 V3.0 补全策略

V3.0 **不改变 V2.0 的 Scheme 架构**，而是在其基础上补充缺失的用户故事和 API 设计，确保每个 Jira 核心功能都有对应的实现。

## 2. Issue Link（任务关联）

### 2.1 功能说明

Jira Issue Link 允许用户在一个 Issue 上关联另一个 Issue，形成需求网络。每个链接有方向性和链接类型。

### 2.2 链接类型（8 种）

| 链接类型 key | 名称 | 方向 | 说明 |
|-------------|------|------|------|
| `blocks` | 阻塞 | 单向 | 当前任务阻塞目标任务 |
| `is_blocked_by` | 被阻塞 | 单向 | 当前任务被目标任务阻塞 |
| `duplicates` | 重复 | 单向 | 当前任务与目标任务重复 |
| `is_duplicated_by` | 被重复 | 单向 | 当前任务被目标任务重复 |
| `relates_to` | 相关 | 双向 | 两个任务相关 |
| `causes` | 导致 | 单向 | 当前任务导致目标任务 |
| `is_caused_by` | 被导致 | 单向 | 当前任务被目标任务导致 |
| `clones` | 克隆 | 单向 | 当前任务克隆自目标任务 |

### 2.3 用户故事

#### US-ISSUELINK-001: 创建任务链接
**作为** 任务负责人，**我希望** 将当前任务与其他任务关联，**以便** 表达任务之间的关系。

验收标准：
- [ ] 在任务详情页点击"添加关联"按钮
- [ ] 选择链接类型（blocks / relates_to / duplicates 等）
- [ ] 搜索并选择目标任务（支持 JQL 搜索）
- [ ] 自动发送通知给被关联任务的负责人
- [ ] 链接记录到任务操作历史

#### US-ISSUELINK-002: 查看任务链接
**作为** 团队成员，**我希望** 查看一个任务的所有链接，**以便** 了解任务之间的关系网络。

验收标准：
- [ ] 在任务详情页"关联"Tab 中显示所有链接
- [ ] 按链接类型分组显示（正向/反向）
- [ ] 点击链接跳转到目标任务
- [ ] 显示链接的创建人和创建时间
- [ ] 支持筛选特定类型的链接

#### US-ISSUELINK-003: 批量管理链接
**作为** 项目经理，**我希望** 批量创建或删除任务链接，**以便** 快速建立任务网络。

验收标准：
- [ ] 支持 CSV 导入批量创建链接
- [ ] 支持批量删除选中的链接
- [ ] 批量操作记录到审计日志

### 2.4 数据库设计

```sql
-- issue_link_type（链接类型）
CREATE TABLE issue_link_type (
  link_type_id   BIGINT PK AUTO_INCREMENT,
  link_key       VARCHAR(32) NOT NULL UNIQUE,  -- blocks / relates_to
  name           VARCHAR(64) NOT NULL,           -- 阻塞 / 相关
  inward         VARCHAR(64),                     -- 反向描述：被阻塞
  outward        VARCHAR(64),                     -- 正向描述：阻塞
  is_system      TINYINT DEFAULT 0,              -- 1=内置不可删除
  is_active      TINYINT DEFAULT 1,
  created_at     DATETIME
);

-- issue_link（任务链接）
CREATE TABLE issue_link (
  link_id        BIGINT PK AUTO_INCREMENT,
  source_issue_id BIGINT NOT NULL,               -- 源任务
  target_issue_id BIGINT NOT NULL,               -- 目标任务
  link_type_id   BIGINT NOT NULL,                -- 链接类型
  created_by     BIGINT NOT NULL,
  created_at     DATETIME,
  FOREIGN KEY (source_issue_id) REFERENCES issue(id),
  FOREIGN KEY (target_issue_id) REFERENCES issue(id),
  FOREIGN KEY (link_type_id) REFERENCES issue_link_type(link_type_id)
);
```

  deleted_by    BIGINT,
  FOREIGN KEY (issue_id) REFERENCES issue(id)

## 4. Time Tracking（时间追踪）

### 4.1 功能说明

Jira Time Tracking 包含三个核心概念：预估工时（Estimate）、剩余工时（Remaining Estimate）、已记录工时（Time Spent）。

### 4.2 用户故事

#### US-TIME-001: 设置预估工时
**作为** 任务负责人，**我希望** 设置任务的预估工时，**以便** 评估任务工作量。

验收标准：
- [ ] 在任务详情页设置预估工时（格式：1w 2d 3h 4m）
- [ ] 支持快速选择预设值（1h / 2h / 4h / 8h / 1d / 3d）
- [ ] 预估工时显示在任务卡片和列表中

#### US-TIME-002: 记录已工时
**作为** 任务负责人，**我希望** 记录在任务上花费的时间，**以便** 追踪实际工作量。

验收标准：
- [ ] 在任务详情页记录工时（工时 + 日期）
- [ ] 支持添加工时说明
- [ ] 记录后自动更新剩余工时 = 预估工时 - 已记录工时
- [ ] 剩余工时为负时给出警告
- [ ] 工时记录支持编辑和删除

#### US-TIME-003: 进度可视化
**作为** 项目经理，**我希望** 查看任务的工时进度，**以便** 了解任务完成情况。

验收标准：
- [ ] 任务卡片显示进度条（已记录/预估）
- [ ] 看板列显示该列任务的总工时
- [ ] 任务列表显示工时列

### 4.3 数据库设计

```sql
-- 在 issue 表中增加时间追踪字段（V3.0）
ALTER TABLE issue ADD COLUMN estimate_seconds BIGINT;         -- 预估工时（秒）
ALTER TABLE issue ADD COLUMN remaining_seconds BIGINT;        -- 剩余工时（秒）
ALTER TABLE issue ADD COLUMN time_spent_seconds BIGINT DEFAULT 0;  -- 已记录工时（秒）

-- worklog（工时记录）
CREATE TABLE worklog (
  worklog_id     BIGINT PK AUTO_INCREMENT,
  issue_id       BIGINT NOT NULL,
  user_id        BIGINT NOT NULL,
  started_at     DATETIME,                             -- 工作开始时间（可选；用户手动指定或系统自动填入）
  duration_seconds BIGINT NOT NULL,                      -- 工时时长（秒）
  description    VARCHAR(512),                       -- 工时说明
  created_at     DATETIME,
  updated_at     DATETIME,
  is_deleted     TINYINT DEFAULT 0,
  FOREIGN KEY (issue_id) REFERENCES issue(id)
);
```

### 5.3.X 工时记录的 started_at 语义说明

**`started_at` 字段定义**：
- `started_at` 为**可选**字段，表示本次工时记录对应的工作开始时间
- 用户记录工时时，只需输入 `duration_seconds`（时长）和 `description`，`started_at` 由系统自动填入当前时间（除非用户手动指定）
- `started_at` 用于工时时间线展示和团队生产力分析，不影响工时统计

**字段设计**：
| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| duration_seconds | BIGINT | NOT NULL | 工时时长（秒），精确到 60（1 分钟）|
| started_at | DATETIME | NULL | 工作开始时间（可选；用户手动指定或系统自动填入）|
| created_at | DATETIME | NOT NULL | 工时录入时间（用于日报/月报统计）|

> **注意**：`started_at` 与 `created_at` 可能不一致（如回补历史工时时，`created_at` 是补录时的时间戳，`started_at` 是实际工作发生的时间）。

## 5. Sub-task（子任务增强）

### 5.1 功能说明

Jira Sub-task 是专门的任务类型，用于将父任务拆分为更小的工作单元。

### 5.2 用户故事

#### US-SUBTASK-001: 创建子任务
**作为** 任务负责人，**我希望** 将一个任务拆分为多个子任务，**以便** 细分工作项。

验收标准：
- [ ] 在父任务详情页点击"添加子任务"
- [ ] 子任务继承父任务的项目、迭代
- [ ] 子任务完成后，父任务可自动感知
- [ ] 子任务可有自己的工作流（可与父任务不同）
- [ ] 子任务可单独分配给不同的人
- [ ] 父任务显示所有子任务的进度汇总

#### US-SUBTASK-002: 子任务 Issue Type
**作为** 项目管理员，**我希望** 为项目配置子任务的 Issue Type，**以便** 限制子任务的类型。

验收标准：
- [ ] 在 Issue Type Scheme 中指定子任务类型（如 Task）
- [ ] 每个父 Issue Type 关联一个子 Issue Type
- [ ] 子任务类型只能作为子任务创建，不能独立创建

### 5.3 数据库设计

```sql
-- 在 issue 表中增加父子关系字段
ALTER TABLE issue ADD COLUMN parent_id BIGINT;              -- 父任务 ID
ALTER TABLE issue ADD COLUMN is_subtask TINYINT DEFAULT 0; -- 是否为子任务
ALTER TABLE issue ADD COLUMN root_id BIGINT;                -- 根任务 ID（用于快速查询整棵任务树）

-- index 优化
CREATE INDEX idx_issue_parent ON issue(parent_id);
CREATE INDEX idx_issue_root ON issue(root_id);
```

## 6. Automation 执行引擎（补全 V2.0）

### 6.1 功能说明

V2.0 已设计 Automation Rule 的数据结构（trigger/condition/action），V3.0 需要实现执行引擎。

### 6.2 执行引擎设计

```
触发事件发生
    │
    ▼
规则引擎扫描所有启用的规则
    │
    ▼
匹配 trigger_type（issue_created / field_changed / scheduled）
    │
    ▼
评估 conditions（AND/OR 条件树）
    │
    ├── 条件不满足 → 跳过
    │
    ▼
执行 actions（按 execute_order）
    │
    ├── SET_FIELD / SEND_NOTIFICATION / CREATE_SUBTASK 等
    │
    ▼
记录执行日志（automation_execution_log）
    │
    ▼
出错则按 fail_action 处理（skip / error）
```

└───────────────────────────────────────────────────────────────────┘
```

**触发器类型**：
- `issue.created` — 任务创建时
- `issue.status_changed` — 任务状态变更时
- `scheduled` — 定时触发（配置 cron 表达式）
- `issue.due_date_approaching` — 截止日期临近时（提前 N 天）

**调度机制**：
- `scheduled` 触发器使用分布式调度（基于 Redis 或数据库锁）
- 调度周期：支持 cron 表达式（每分钟精度）
- 多个规则同时触发时，按规则创建时间顺序执行

│      BUG · 有 2 个子任务、3 个评论                            │
│ ──────────────────────────────────────────────────────────────── │
│ 已选 0 个                       永久删除选中的任务              │
│ ──────────────────────────────────────────────────────────────── │
│ ⚠️ 回收站数据保留 30 天，超期后自动永久删除                    │
└─────────────────────────────────────────────────────────────────┘
```

**交互规则**：
- 恢复操作将任务还原到原位置（若父任务也已删除则还原到根级）
- 有子任务的任务被删除时，所有子任务一并移入回收站
- 恢复时若原项目已归档，先恢复项目再恢复任务
- 永久删除后数据无法恢复，需二次确认

### 6.5 触发器调度机制（L-04 部分补全）

#### 定时触发器（scheduled）

定时触发器使用分布式调度器（基于数据库乐观锁或 Redis 分布式锁）确保多实例环境下不重复执行。

**Cron 表达式配置**：
```
scheduled: "0 0 8 * * ?"  // 每天 08:00 执行
scheduled: "0 0/30 * * * ?" // 每 30 分钟执行
scheduled: "0 0 9 ? * MON" // 每周一 09:00 执行
```

**调度流程**：
1. 每个节点上的调度器定时查询 `automation_rule` 表中 `trigger_type = 'scheduled'` 且 `next_run_at <= now()` 的规则
2. 获得锁的节点执行规则，其他节点跳过
3. 执行完成后更新 `last_run_at` 和 `next_run_at`

**优先级**：当同一时刻有多个规则触发时，按 `priority DESC` 顺序执行。

#### Webhook 触发器（webhook）

Webhook 触发器接收外部系统回调：

```
POST /api/v1/automation/webhook/{webhookToken}
X-Webhook-Secret: {签名}
Content-Type: application/json

{
  "event": "custom_event",
  "data": { ... }
}
```

**安全**：每个 Webhook 触发器有独立的 `webhookToken`，用于认证回调来源。

## 13. 数据库补充设计

本节补充 V3.0 功能所需的 5 个核心数据表设计。

### 12.1 sprint（Sprint 迭代）

Sprint 是 Scrum 敏捷开发的核心实体，定义一个时间boxed的迭代周期。

```sql
CREATE TABLE sprint (
  sprint_id        BIGINT PRIMARY KEY AUTO_INCREMENT,
  project_id       BIGINT NOT NULL,
  sprint_name      VARCHAR(128) NOT NULL,
  goal             TEXT,                             -- Sprint 目标描述
  status           VARCHAR(16) NOT NULL DEFAULT 'planning',  -- planning / active / completed / cancelled
  start_date       DATE,                             -- Sprint 计划开始日期
  end_date         DATE,                             -- Sprint 计划结束日期
  actual_start_date DATE,                            -- 实际开始日期（第一张卡移入时记录）
  actual_end_date   DATE,                             -- 实际结束日期（Sprint 完成时记录）
  velocity_plan    INT DEFAULT 0,                    -- 计划速率（故事点总和）
  velocity_actual  INT DEFAULT 0,                    -- 实际速率（Sprint 完成时计算）
  total_story_points DECIMAL(5,1) DEFAULT 0,       -- 计划故事点总数
  completed_story_points DECIMAL(5,1) DEFAULT 0,    -- 已完成故事点
  burndown_snapshot_enabled TINYINT DEFAULT 1,       -- 是否启用燃尽图快照
  created_at       DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at       DATETIME ON UPDATE CURRENT_TIMESTAMP,
  FOREIGN KEY (project_id) REFERENCES project(id)
);
```

**业务规则**：
- `status` 流转：`planning` → `active` → `completed` / `cancelled`
- Sprint 激活时（第一张卡移入）自动记录 `actual_start_date`
- Sprint 完成时自动计算 `velocity_actual = sum(completed_issue.story_point)`
- 一个项目同时只能有 1 个 `active` 状态的 Sprint

### 12.2 cfd_snapshot（Cumulative Flow Diagram 快照）

CFD 是燃尽图的互补图表，记录每个状态在每天结束时累积的任务数量，用于分析工作流效率。

```sql
CREATE TABLE cfd_snapshot (
  snapshot_id    BIGINT PRIMARY KEY AUTO_INCREMENT,
  project_id    BIGINT NOT NULL,
  sprint_id    BIGINT,                              -- NULL 表示项目级（非 Sprint）
  snapshot_date DATE NOT NULL,                       -- 快照日期（每天一个快照）
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,

  -- 各状态的任务数量（每天结束时统计）
  backlog_count      INT DEFAULT 0,                -- 待办
  in_progress_count INT DEFAULT 0,                 -- 进行中
  review_count      INT DEFAULT 0,                  -- 评审中
  done_count        INT DEFAULT 0,                  -- 已完成
  blocked_count     INT DEFAULT 0,                  -- 阻塞中

  UNIQUE KEY (project_id, sprint_id, snapshot_date),
  FOREIGN KEY (project_id) REFERENCES project(id),
  FOREIGN KEY (sprint_id) REFERENCES sprint(sprint_id)
);
```

**采集策略**：
- 每天凌晨 00:05 执行定时任务采集一次
- 当任务状态变更时触发异步更新（乐观更新，不阻塞主事务）
- 保留最近 365 天快照，超期自动归档到冷存储

### 12.3 velocity_sprint（Sprint 速率统计）

记录每个 Sprint 的速率数据，用于预测未来 Sprint 的交付能力。

```sql
CREATE TABLE velocity_sprint (
  id            BIGINT PRIMARY KEY AUTO_INCREMENT,
  sprint_id     BIGINT NOT NULL,
  project_id    BIGINT NOT NULL,
  
  -- 计划数据
  planned_story_points DECIMAL(5,1) DEFAULT 0,   -- 计划故事点
  
  -- 完成数据
  completed_story_points DECIMAL(5,1) DEFAULT 0,  -- 实际完成故事点
  
  -- 完成质量
  completed_count    INT DEFAULT 0,                -- 完成的任务数（不含子任务）
  added_during_sprint INT DEFAULT 0,              -- Sprint 中新增的任务数
  removed_from_sprint  INT DEFAULT 0,            -- Sprint 中移除的任务数
  
  -- 时间
  committed_count  INT DEFAULT 0,                 -- 承诺任务数
  predictability  DECIMAL(4,2) DEFAULT 0,         -- 可预测性 = 实际完成 / 计划（百分比）
  
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (sprint_id) REFERENCES sprint(sprint_id),
  FOREIGN KEY (project_id) REFERENCES project(id),
  UNIQUE KEY (sprint_id)
);
```

### 12.4 issue_cycle_time（任务周期时间统计）

记录每个任务从开始到完成所经历的各阶段时间，用于计算 Lead Time / Cycle Time。

```sql
CREATE TABLE issue_cycle_time (
  id              BIGINT PRIMARY KEY AUTO_INCREMENT,
  issue_id        BIGINT NOT NULL,
  project_id      BIGINT NOT NULL,
  sprint_id       BIGINT,                          -- 当前 / 最后一个 Sprint
  
  -- 各阶段时间戳
  created_at      DATETIME NOT NULL,               -- 任务创建时间（= Cycle Time 起点）
  first_in_progress_at DATETIME,                  -- 首次进入"进行中"状态的时间
  first_review_at DATETIME,                       -- 首次进入"评审"状态的时间
  done_at        DATETIME,                       -- 首次进入"完成"状态的时间（= Cycle Time 终点）
  
  -- 各阶段耗时（秒）
  backlog_seconds  BIGINT DEFAULT 0,             -- 等待时间（创建 → 首次开始）
  in_progress_seconds BIGINT DEFAULT 0,          -- 开发时间（首次开始 → 首次评审）
  review_seconds   BIGINT DEFAULT 0,              -- 评审时间（首次评审 → 完成）
  total_cycle_time_seconds BIGINT DEFAULT 0,     -- 总周期时间（创建 → 完成）
  lead_time_seconds BIGINT DEFAULT 0,            -- 交付时间（含在看板中等其他等待）
  
  -- 计算字段（当 done_at 非空时自动计算）
  computed_at    DATETIME,
  
  FOREIGN KEY (issue_id) REFERENCES issue(id),
  FOREIGN KEY (project_id) REFERENCES project(id),
  FOREIGN KEY (sprint_id) REFERENCES sprint(sprint_id),
  UNIQUE KEY (issue_id)
);
```

**计算触发时机**：
- 任务状态变更为"进行中"时记录 `first_in_progress_at`
- 任务状态变更为"评审中"时记录 `first_review_at`
- 任务状态变更为"完成"时触发所有字段计算并写入

### 12.5 deleted_issue（软删除回收站）

记录所有软删除的任务，用于恢复和审计。

```sql
CREATE TABLE deleted_issue (
  id              BIGINT PRIMARY KEY AUTO_INCREMENT, -- 复用原始 issue.id 作为主键
  original_id     BIGINT NOT NULL,                  -- 原始任务 ID（删除前的 ID）
  project_id      BIGINT NOT NULL,
  
  -- 软删除前的完整快照
  snapshot_data   JSON NOT NULL,                   -- 删除时的完整任务 JSON（含所有字段）
  
  -- 删除元信息
  deleted_by      BIGINT NOT NULL,                 -- 执行删除的用户
  deleted_at      DATETIME NOT NULL,                -- 删除时间
  delete_reason   VARCHAR(32),                     -- manual / project_deleted / bulk_delete / admin
  original_title  VARCHAR(512) NOT NULL,          -- 删除前的标题（用于搜索）
  
  -- 恢复信息
  restored_at     DATETIME,                       -- 恢复时间（NULL 表示未恢复）
  restored_by     BIGINT,                         -- 执行恢复的用户
  is_restored     TINYINT DEFAULT 0,             -- 是否已恢复
  
  -- 永久删除信息（回收站清除时）
  purged_at       DATETIME,                       -- 永久删除时间（NULL 表示仍在回收站）
  
  INDEX idx_deleted_by (deleted_by),
  INDEX idx_deleted_at (deleted_at),
  INDEX idx_original_title (original_title(128)),
  INDEX idx_is_restored (is_restored),
  INDEX idx_purged_at (purged_at)
);
```

**保留策略**：
- 回收站默认保留 30 天，超期自动永久删除
- 管理员可在管理后台配置保留天数（7-365 天）
- `purged_at IS NULL AND is_restored = 0` → 在回收站中
- 恢复任务时生成新 ID，原 `deleted_issue` 记录标记 `is_restored = 1`

## 15. 任务导入与导出（Import / Export）

### 14.1 功能说明

任务导入导出支持从 Excel/CSV 批量迁移数据，以及将任务列表导出为 Excel 离线查看或上报。

### 14.2 数据库设计

```sql
-- 导入任务记录
CREATE TABLE issue_import (
  id              BIGINT PK AUTO_INCREMENT,
  import_id       VARCHAR(36) NOT NULL,
  project_id      BIGINT NOT NULL,
  creator_id      BIGINT NOT NULL,
  file_name       VARCHAR(255) NOT NULL,
  file_url        VARCHAR(500),
  total_rows      INT,
  success_count   INT DEFAULT 0,
  fail_count      INT DEFAULT 0,
  status          VARCHAR(20),
  created_at      DATETIME,
  finished_at     DATETIME,
  FOREIGN KEY (project_id) REFERENCES project(id),
  FOREIGN KEY (creator_id) REFERENCES user(id)
);

CREATE TABLE issue_import_error (
  id              BIGINT PK AUTO_INCREMENT,
  import_id       VARCHAR(36) NOT NULL,
  row_number      INT,
  field_name      VARCHAR(100),
  field_value     VARCHAR(500),
  error_message   VARCHAR(500),
  created_at      DATETIME
);

CREATE TABLE issue_export (
  id              BIGINT PK AUTO_INCREMENT,

);
```

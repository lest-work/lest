### 统一的通知事件总线

通知模块作为统一的事件总线，接收所有业务模块的事件并分发通知。共定义 70+ 个跨模块通知事件，详见 `通知与消息.md` 文档第 3.3 节。其中 AI 服务相关事件包括：AI_REVIEW_COMPLETED（AI 代码审查完成）、AI_SUMMARY_GENERATED（AI 总结生成）、AI_TASK_SUGGESTIONS_READY（AI 任务建议就绪）、AI_INSIGHT_GENERATED（AI 洞察生成）等。

## 扩展生态体系

LEST Platform 的核心差异化能力在于**开放性**。通过插件系统和开放平台两大支柱，构建完整的扩展生态：

```
                    ┌─────────────────────────────────────────────┐
                    │           LEST Platform 核心               │
                    │  (任务、项目、会议、代码、CI、发布、绩效、编码追踪) │
                    └─────────────────┬───────────────────────┘
                                      │
                    ┌─────────────────┴───────────────────────┐
                    │                  │                        │
              ┌─────▼──────┐   ┌──────▼──────┐      ┌─────▼──────┐
              │   插件系统   │   │   开放平台   │      │   AI 服务   │
              │   Plugin    │   │  Open API   │      │  AI        │
              │   System    │   │  Platform   │      │  Service   │
              └─────────────┘   └────────────┘      └─────────────┘
                     │                  │                     │
              ┌──────▼──────┐   ┌──────▼──────┐      ┌──────▼──────┐
              │   插件 SDK   │   │  REST API   │      │ AI Provider │
              │ TypeScript  │   │  Webhook    │      │   扩展点    │
              │ Java        │   │  OAuth2     │      │             │
              └─────────────┘   └────────────┘      └─────────────┘
```

### 插件系统

通过插件系统，开发者可以：
- 在平台特定位置注入 UI 组件（自定义 Tab、侧边栏、按钮）
- 扩展业务实体（任务自定义字段、项目模板）
- 添加定时任务（定期同步外部数据）
- 接入自定义 AI Provider（新模型、新能力）
- 实现第三方集成（Jira、飞书、钉钉）

详见 `插件系统.md`。

### 开放平台

通过开放平台，外部系统可以：
- 通过 REST API 操作平台所有业务数据
- 订阅 50+ 业务事件（Webhook）
- 实现 OAuth2 授权登录
- 生成个人访问令牌进行 API 调用
- 集成到 CI/CD 流水线、自动化工具

详见 `开放平台.md`。

### WakaTime 集成

通过 WakaTime 集成，平台实现开发者编码活动的自动追踪：
- 兼容 WakaTime 官方 API，IDE 插件直接对接
- 自动记录编码时长、项目分布、语言使用
- 心跳数据自动关联到具体任务
- 为团队绩效提供编码活跃度数据

详见 `WakaTime集成.md`。

## 开发计划

各模块的开发优先级如下：

> **重要更新（V2.0）**：LEST Platform 是 **AI-Native 敏捷管理平台**，AI 不是独立的"后期功能"，而是从 Phase 2 开始就与各业务模块并行开发的**中央智能引擎**。AI 服务通过 API 和事件机制被各业务模块调用，为 WakaTime 数据、任务数据、代码数据等赋予智能分析能力。

### 阶段一：基础服务（核心 MVP）

| 服务 | 模块 | 优先级 | 预估工作量 | AI 关联 |
|------|------|--------|-----------|---------|
| lest-gateway | API 网关 | P0 | 1周 | - |
| lest-auth | 认证服务 | P0 | 2周 | - |
| lest-system | 系统管理服务 | P0 | 2周 | - |
| lest-project | 项目管理服务 | P0 | 2周 | - |
| lest-task | 任务管理服务 | P0 | 3周 | 任务完成时触发 AI 分析 |

### 阶段二：协作服务 + AI 引擎（并行启动）

| 服务 | 模块 | 优先级 | 预估工作量 | AI 关联 |
|------|------|--------|-----------|---------|
| lest-wakapi | WakaTime 集成服务 | P0 | 2周 | 编码数据供 AI 分析 |
| lest-notification | 消息通知服务 | P0 | 2周 | AI 洞察推送通道 |
| lest-code | 代码管理服务 | P0 | 2周 | MR 创建触发 AI 审查 |
| **lest-ai** | **AI 服务（引擎核心）** | **P0** | **3周** | **中央智能引擎** |

### 阶段三：流程服务

| 服务 | 模块 | 优先级 | 预估工作量 | AI 关联 |
|------|------|--------|-----------|---------|
| lest-meeting | 会议管理服务 | P0 | 1周 | 会议结束触发 AI 总结 |
| lest-performance | 团队绩效服务 | P0 | 2周 | 深度调用 AI 绩效洞察 |
| lest-ci | CI/CD 服务 | P1 | 2周 | 构建失败触发 AI 分析 |
| lest-release | 发布管理服务 | P1 | 2周 | - |

### 阶段四：AI 深度应用 + 扩展服务

| 服务 | 模块 | 优先级 | 预估工作量 | AI 关联 |
|------|------|--------|-----------|---------|
| **lest-ai（增强）** | AI 绩效洞察完整版 | P0 | 2周 | 全模块数据整合 |
| lest-plugin | 插件系统 | P1 | 3周 | AI Provider 扩展点 |
| lest-open | 开放平台 | P1 | 3周 | AI 能力对外暴露 |
| lest-search | 搜索服务 | P2 | 2周 | AI 语义搜索 |
| lest-file | 文件服务 | P1 | 1周 | - |

### AI 引擎开发里程碑

```
Phase 1 末期
  │
  ├── AI 网关核心（统一调用接口、多模型路由、降级策略）
  ├── 编码活跃度分析 API（WakaTime 数据接入）
  └── AI 代码审查 API（MR 自动审查）
          │
          ▼
Phase 2 末期
  ├── AI 绩效洞察 API（工时预估评估、个人发展建议）
  ├── AI 团队状态报告 API（定时生成）
  ├── AI 项目风险预警 API（多维度风险识别）
  └── AI 提示词管理和成本控制
          │
          ▼
Phase 3
  └── AI 深度应用（个人发展建议精细化、跨模块关联分析）
```

## 文档编写规范

1. 所有 PRD 放在 `PRD/V{版本}/` 目录下
2. 每个功能一个独立的 Markdown 文件
3. 文档使用中文编写
4. 文档命名格式：`{功能名称}.md`
5. 跨模块接口需在各相关 PRD 中同时定义
6. 新增错误码需在 README 中更新分配表
7. 新增扩展点机制需在插件系统 PRD 中定义接口
8. 数据库设计统一在 DATABASE.md 中维护
9. 系统架构统一在 ARCHITECTURE.md 中维护

## 贡献指南

1. 新功能开发前，先编写 PRD
2. PRD 评审通过后，再开始开发
3. 跨模块功能需在相关 PRD 中同步更新
4. 开发过程中 PRD 如有变更，需要同步更新文档
5. 文档更新需要通过 Code Review
6. 扩展功能优先通过插件系统实现，而非修改核心代码
7. 数据库表需在 DATABASE.md 中有完整的 CREATE TABLE 定义
8. API 接口需在 API.md 中有完整定义
